# CLAUDE.md

Canonical context for AI assistants working in this repo. If anything elsewhere disagrees, trust this file.

## What this is
**ORA Navigator** — an AI assistant + proposal toolkit for Morgan State University's **Office of
Research Administration** (ORA): grants, compliance, pre/post-award, forms, ORA contacts, and a
guided workspace that helps inexperienced PIs actually write and submit fundable proposals.

## Three services
| Service | Stack | Local port | Container port |
|---|---|---|---|
| **frontend** | Vite + React 19 (PWA) | 3001 | 8080 |
| **backend** | FastAPI, **single-worker uvicorn** | 5002 | 5000 |
| **adk_agent** | Google ADK + Gemini 2.5 Flash (Vertex AI), `VertexAiSearchTool` over one unified KB datastore | 8081 | 8080 |

The chat path: frontend → backend (`vertex_agent.py`) → ADK agent (`adk_agent/ora_navigator_unified/`) → Gemini on Vertex AI. The backend also serves the proposal toolkit (below) and talks to Cloud SQL (MySQL) + Redis.

## Build / test / deploy
- **Backend tests (keep green before any push/deploy):**
  ```bash
  cd backend && JWT_SECRET=test-secret TRUSTED_HOSTS=testserver,localhost,127.0.0.1 \
    python3 -m pytest -q --ignore=tests/test_agent_instruction.py
  ```
  Env vars are required because importing `main` constructs the app. `test_agent_instruction.py` needs `google-adk` (not always installed locally). App-level e2e tests use FastAPI `TestClient` with `dependency_overrides` for `get_db` / `get_current_user` (see `tests/test_proposals_api_e2e.py`).
- **Frontend:** `cd frontend && npm run build` (no `node_modules`? `npm install` first). It's a **PWA** — verify changes in a fresh/incognito window.
- **Deploy:** pushes to **`main`** trigger **Cloud Build** (`cloudbuild.yaml`) which builds + deploys all three Cloud Run services in project `infra-vertex-494621-v1`, region `us-central1`. There's also a manual `deploy-cloudrun.sh` (needs local `gcloud`). The **backend** runs `--min-instances=1 --cpu-boost` so the chat path never pays a 30–60s cold start (the dominant cause of slow first answers); the frontend stays `min-instances=0` (static, off the per-question path). **The ADK service MUST stay a SINGLE instance (`--min-instances=1 --max-instances=1`).** It stores agent sessions **in memory** (no `ADK_SESSION_DB_URI`), so with >1 instance a session created on one instance **404s on `/run_sse`** when the next call is routed to another — the user sees *"An error occurred while processing your question"*, intermittently (random per-instance routing). A brief bump to `--min-instances=2` for concurrency caused exactly this outage. **Never scale ADK horizontally without wiring a shared session store (`ADK_SESSION_DB_URI`) first.** `concurrency=80` lets one warm ADK instance handle normal + demo load fine. (You can apply a scaling change live without a full rebuild via `gcloud run services update <svc> --min-instances=N --max-instances=M`.) **`cloudbuild.yaml` is a single point of failure** — the backend deploy step is an inline bash heredoc, so a one-character typo there (e.g. a stray `=` swallowing `--min-instances`) breaks the *whole* deploy and silently blocks every merge from shipping. Validate it (`python3 -c "import yaml; yaml.safe_load(open('cloudbuild.yaml'))"`) after any edit.
- **Backend service-account IAM** (`oranavigator-backend@<proj>.iam.gserviceaccount.com`): the admin dashboard **writes** to the KB datastore, so the SA needs **`roles/discoveryengine.editor`**. With `roles/discoveryengine.viewer` alone (what `deploy-cloudrun.sh setup_iam` used to grant) reads work — the KB browser lists and searches fine — but every **Save 403s** with `discoveryengine.documents.update denied`, which reads as a UI bug rather than a permissions one. Editor covers `documents.create/.update/.delete`. Note this is a **project-level** grant and therefore broader than needed (it also allows datastore/engine create+delete); least privilege would be a custom role limited to `discoveryengine.documents.*`. Accepted deliberately — revisit if the project gains other Discovery Engine resources. `roles/discoveryengine.viewer` is now redundant alongside it. **`gcloud` may not be on `PATH` locally** — it lives at `~/google-cloud-sdk/bin/gcloud`.
- **Secrets** live in Secret Manager as `ora-database-url`, `ora-jwt-secret`, `ora-admin-email`, `ora-admin-password`, `ora-redis-url`, `ora-firecrawl-api-key` (mapped in `cloudbuild.yaml`). The Cloud Build runs as the **default compute service account** (`<projnum>-compute@…`), which lacks `secretmanager.secrets.create` — so the `deploy-backend` step **self-heals** `ora-firecrawl-api-key` (creates a `unset` placeholder if missing) so a missing optional secret never breaks the deploy. Create real secret values via the Console (the project picker prevents the "wrong project" mistake).
- **`GOOGLE_GENAI_USE_VERTEXAI=TRUE`** must be set on the ADK **and** backend services (in `cloudbuild.yaml`) or Gemini falls back to the developer API and every call fails with "No API key was provided."

## Golden rules (followed everywhere; keep following them)
1. **AI is advisory; the deterministic core is authoritative.** Numbers, pass/fail, and eligibility verdicts are computed by code — the LLM only explains, drafts prose, outlines, or gives advisory feedback. It can never change a figure.
2. **Ground every AI claim.** Any "covered/strong" finding must quote the source text; unquotable claims are dropped (`_verify_evidence` in `draft_critic.py` / `section_coach.py`). The membership check **collapses whitespace on both sides** (`" ".join(s.split())`) — a pasted/hard-wrapped draft has newlines where Gemini's quote has spaces, and a raw substring match would wrongly reject a real quote and mark a complete section "NOT FOUND".
3. **Graceful fallback.** Every AI path returns a deterministic result when Gemini is unavailable (`gemini_client.generate_json/text` return `None` → caller falls back). Never error because the model is down.
4. **Human reviews before commit.** Extractions and budgets are returned for review; nothing is saved until the user confirms.
5. **Conventions:** structured data stored as **TEXT columns + `json.dumps/loads`** (e.g. `Submission.budget_json`, `compliance_json`, `sections_json`, `notes`). New columns are added with a **self-healing migration** in `main.py:init_db()` (try `SELECT col`; on error `ALTER TABLE ADD COLUMN`) — safe on prod MySQL, no Alembic.
6. **One feature = one focused change.** Don't refactor unrelated code. Verify before claiming done.
7. **Never push automatically.** Do NOT run `git push` (and don't open PRs) on your own — only when the user explicitly asks in that message. Pushing to `main` triggers the Cloud Build deploy, so an unrequested push ships to production. Committing locally is fine to stage work, but wait for an explicit "push"/"deploy" before it leaves the machine. Approval to push once does not carry over to later changes.

## The Proposal Toolkit (backend `services/` + per-proposal modals in `frontend/src/components/`)
Each proposal (`Submission` + `SubmissionTask` in `models.py`, logic in `proposals_service.py`, sponsor task templates in `proposal_templates.py`) has a toolbar of tools:

- **Solicitation ingestion** — `solicitation_extractor.py` (PDF→text via pdfplumber → Gemini → contract dict with `source_quotes`; fabricated quotes flagged via `_verify_source_quotes`). Two entry points: upload a PDF, or **paste a URL** (`url_fetcher.py`). The URL fetcher is SSRF-guarded (resolve + reject private/loopback/link-local IPs, pin the validated IP to defeat DNS rebinding). Some funder sites (NSF/Akamai) **block cloud-datacenter IPs**, so a direct fetch falls back through a scraper chain: **Firecrawl** (`_fetch_via_firecrawl`, preferred — runs a real browser from non-blocked IPs; needs `FIRECRAWL_API_KEY`, and treats the `unset` deploy-placeholder as no-key), then `r.jina.ai` (set `JINA_API_KEY`), then `allorigins`. Uploaded PDFs are 100% local (pdfplumber) and never hit an IP block. `create_submission_from_solicitation` seeds the checklist; multi-category deadlines are surfaced via `deadline_details`. The extracted **eligibility text** is stored in `notes` and fed to the Drafting Coach.
- **Budget Helper** — `budget_helper.py`: deterministic F&A/MTDC/fringe math (Morgan's real rates), sponsor-cap check. **Coaching layer:** `budget_advisories()` (sanity flags), `suggest_trims()` (concrete cuts to get under cap), category/F&A/fringe tooltips. **v2:** multi-year projection with escalation (`compute_budget` detects `project_years`; single-year output unchanged), CSV export (`budget_to_csv` + `/budget.csv`), per-line justification, and a **deterministic prose** budget justification (`draft_justification` — labeled paragraphs, reads like a real submission). AI only polishes the prose and is forbidden to change figures; the `/api/budget/justification` endpoint falls back to the template when the AI output is empty **or truncated** (missing the total-cost figure — a Gemini-under-load failure mode). The UI **effort→person-months helper is read-only**, derived from each person's `effort_pct` in the People rows (not a separate input), so it can never disagree with the budget. **Per-category caps:** when a solicitation defines multiple proposal categories with different award maxima (e.g. NSF IDSS Category I/II/III), `solicitation_extractor` returns `budget_cap_details` (`[{category, cap}]`, a stated **range → max**), persisted as a `Category caps:` line in `notes` (round-tripped by `reconstruct_solicitation_context`; category names sanitized of `;`/em-dash); the single `budget_cap` stays the smallest. The modal shows a **Funding category** picker that fills the Sponsor cap — default is a blank "Select your category…" so the PI consciously picks; the picker appears only for 2+ categories. **Spreadsheet view:** `compute_budget` also returns a render-ready `table` (`{columns, rows}`; single-year = one `Amount` column, multi-year = a column per year + `Total`) built by `_budget_table` from the per-year breakdowns (now retained instead of discarded). The modal renders it as a read-only **Budget table** grid — it formats numbers but computes none, so it always matches the summary and the CSV (row order mirrors `budget_to_csv`). **Modal layout:** the Budget table's `<table>` uses class **`bh-table`** — NOT `bh-grid`, which is the form's two-column `display:grid` and silently turned the table into a broken 2-column grid (thead/tbody split, amounts clipped). The whole modal body (form + summary + table + justification) scrolls as **one** `.bh-scroll` region; the navy **Summary is `position:sticky`** so the total stays in view while the taller form scrolls (disabled on mobile). The **justification draft panel has a Close (×)** that clears `justification`+`perLine` without closing the whole modal. **v3 (multi-year coaching):** the multi-year path is otherwise unchanged but now *explains itself* to a first-time PI — `ESCALATION_GUIDANCE` tooltip (escalation hits **salaries only**, ~2-3% federal norm; exposed via `rate_options()`), per-year **salary deltas** in `multi_year.years` (`salary_delta`/`salary_delta_pct`, shown as "salary +$X (Y%)"), the cap badge spells out it's the **cumulative N-year TOTAL** (`multi_year.cap_basis="cumulative"`), `draft_justification` adds a **"Period of performance"** paragraph (figures from `multi_year`; placed *before* the final total so the endpoint truncation guard still trips), and `multi_year_advisories()` flags **0% escalation** (info) / **>10%** (warn) — appended to `advisories` at the `compute_budget` level since `_compute_single` never sees `escalation_pct`. All additive/deterministic; AI still only polishes. Out of scope (deliberately): per-person escalation rates and ramp-up/down spending profiles.
- **Drafting Coach** — `section_coach.py`: per-section **outline** + advisory **feedback** on the PI's own draft (covered/missing elements, grounded). Coaching only — never writes prose. Section menu is **by sponsor** (`_SPONSOR_ORDER`: NSF → Project Summary / Project Description / Broader Impacts / Data Management Plan; NIH → Specific Aims / Research Strategy / DMP; else generic). **v2:** save drafts (`sections_json`), live length meter (`WORD_TARGETS`), "match THIS solicitation" (feeds required attachments/eligibility/page limits), deterministic `clarity_check()`. **v3 (solicitation-aware):** `outline_section` now USES the solicitation context it already received — outline tips reflect this funder's page limits/required attachments/eligibility, tailoring runs on **topic OR context** (not just topic), and the "match THIS solicitation" panel shows in **outline** mode too; `SECTION_SAMPLES` maps each section to an authored **Sample Proposals Library** entry (validated via `get_sample` so a link can't 404), surfaced as a "See a worked example" link in both modes; `review_section` (via `_REVIEW_SYSTEM`) also flags solicitation requirements the draft doesn't address (evidence-grounding unchanged). **v4 (reviewer-lens + cross-section coherence):** **(A)** a deterministic per-sponsor `REVIEW_RUBRICS` / `review_rubric(sponsor)` — the panel's actual scoring criteria (NSF Intellectual Merit + Broader Impacts; NIH Significance/Innovation/Approach/Investigators/Environment; generic fallback) — is attached to **both** outline and review results (so the PI writes TO the rubric), and `review_section` asks for advisory `reviewer_notes` (one per criterion, "how a panel would judge THIS draft"). Notes are **opinions/questions, NOT coverage claims** (no quote required); coverage stays in the grounded `checklist`. Notes are filtered to valid criteria; empty when AI is off (rubric always present). **(B)** `coherence_check(sponsor, drafts, context, budget)` — a NEW advisory check that compares the PI's *saved* sections against each other: NIH Specific Aims ↔ Research Strategy, scope ↔ eligibility, timeline ↔ budget (`_coherence_candidate_pairs` builds only the pairs whose source sections exist). Every `"aligned"` verdict must quote **both** sides, verified with the shared `_quote_in()` helper (the whitespace-collapse match, **refactored out of `_verify_evidence`** — golden rule 2); unverifiable → demoted to `"unclear"`, quote dropped. Deterministic fallback when AI is off (incl. an Aim 1/2/3 label co-occurrence pass; never fabricates quotes). `ready:false` when <2 sections saved or no pair applies. Endpoint **`POST /api/me/submissions/{id}/coherence`** (reads existing `sections_json`/`budget_json`/`notes` — no new state); UI is a third **"Cross-section"** tab gated on ≥2 saved drafts, plus a `ReviewerPanel` rendered in outline+review. **v5 (solicitation responsiveness matrix):** `responsiveness_matrix(sponsor, drafts, context)` — a NEW whole-proposal advisory check that maps each explicit solicitation **ask** onto the PI's *saved* drafts, attacking the #1 triage reason (non-responsiveness). The requirements list is assembled **deterministically** by `_responsiveness_requirements` from already-grounded state — page-limited named sections (`context["page_limits"]` keys → `SECTIONS` labels), required attachments/elements (`context["required_attachments"]`), the sponsor's `review_rubric()` criteria, and eligibility text — **so the AI can never invent an ask** (it receives a fixed `id` set and only assigns coverage). Status vocabulary is `addressed | partial | not_found | check_by_hand` (draft_critic's `missing|unclear` renamed) — **coverage only, never a score/verdict** (respects the removed-Fundability boundary below). Every `addressed`/`partial` must quote a saved draft verbatim, verified with `_quote_in`; unverifiable → demoted to `check_by_hand`, quote dropped, and `where` is re-attributed authoritatively to the section containing the verified quote (not trusted from the model). Deterministic fallback marks every row `check_by_hand` (never guesses `not_found` — offline it can't prove absence — and never fabricates a quote). Dedupe by normalized label (DMP as both attachment + page-limited section collapses to one, section wins). Endpoint **`POST /api/me/submissions/{id}/responsiveness`** (reads `sections_json`/`notes` — no new state, no budget); UI is a fourth **"Responsiveness"** tab gated on ≥1 saved draft. **UI removed (2026-07):** the **Cross-section** (v4 coherence) and **Responsiveness** (v5) tabs were removed from the Drafting Coach UI by product decision — the Coach now offers only "Outline it" and "Feedback on my draft". Both **backend endpoints (`/coherence`, `/responsiveness`) and services remain** (now unused by the UI); re-add the tabs to restore them.
- **Draft Critic vs Drafting Coach** — different jobs: the **Coach** grades the *writing* of one section; the **Critic** (`draft_critic.py`) checks the *whole package's completeness/compliance* (required sections & **solicitation-required attachments**, page limits, budget vs cap) + advisory AI review. Completeness is the Critic + the solicitation checklist, never the Coach.
- **Guided steps** — `task_guidance.py`: keyword-matched how-to + sample per checklist task, attached in `_submission_task_to_dict`; per-task "add to calendar".
- **Guided Proposal Pathway** (the proposal detail view in `MyProposals.jsx`, presentation/orchestration — **no new tool logic**) — for inexperienced PIs, the flat toolbar is a **staged path**: Set up → Build → Review & submit, each tool carrying a ✓ status badge from serializer flags (`has_budget`/`has_compliance`/`has_sections` = `bool(sections_json)`; Solicitation from the `hasSolicitation` notes-regex; Critique is stateless). A pure `nextStep(submission)` picks the single recommended action — only that button gets the accent (the rest share one calm style) — and a plain-language **"What to do next"** card explains the current step + why (active proposals only). Delete + Add-to-Calendar live in a `⋯` overflow. The summary shows the **internal ORA routing deadline** (`proposals_service.internal_routing_deadline` → `internal_deadline` field, **5 business days before** the sponsor date, weekends skipped) so a first-timer plans backward from the real institutional cutoff.
- **Draft Critic** (`draft_critic.py`) — mechanical pre-submission checks (page limits, required sections/attachments, budget vs cap) + advisory AI review.
- **Compliance Sentinel** (`compliance_sentinel.py`) — IRB/IACUC/COI/RCR/export rules. **Deadline Watcher** (`deadline_watcher.py`) — AI-personalized reminder emails (cron). `ics_export.py` — calendar feeds.

**Removed:** the former **Fundability / Eligibility self-check** tool (reviewer-lens scoring + go/no-go) was deleted per product decision — `services/fundability.py`, `services/eligibility.py`, their tests, the `FundabilityModal`, and the `/fundability/criteria` · `/eligibility` · `/fundability` endpoints are gone. Do not re-add. The eligibility *text* extracted from the solicitation is still kept (`_eligibility_text_from_notes`) and fed to the Drafting Coach.

## Opportunity Finder + Fit Advisor (standalone page, not a per-proposal tool)
A discovery surface at **`/opportunities`** (navbar "Find Funding" button → `frontend/src/components/OpportunityFinder.jsx`, styled like the Samples/Forms surfaces) that closes the gap *before* a PI has a solicitation: they describe their research in plain text and get a ranked list of **live, open federal opportunities**, each with a grounded fit explanation, an eligibility verdict, and a mechanism note. Logic in **`services/opportunity_finder.py`**; endpoint **`POST /api/opportunities/search`** (auth'd; enriches the query with the user's saved interests from `UserMemory`).
- **Source = Grants.gov public REST API** (fixed host `api.grants.gov`, no key, covers all 26 federal agencies — NSF/NIH/DOE/…): `search2` (keyword → opp ids/basics) then `fetchOpportunity` per top result (the rich synopsis: `applicantTypes`, `applicantEligibilityDesc`, `synopsisDesc`, award ceiling, close date, contact). Foundations are out of v1 (no free API). Returns `[]` (never fabricates) on API failure.
- **Eligibility = deterministic STOP-gate** (`eligibility_gate`): Morgan State's baked-in profile (public IHE / HBCU / MSI — like the F&A rates are baked in) matched against the opportunity's applicant-type *descriptions* → `eligible` / `unrestricted` / `see_text` / `ineligible`. AI advisory only for the free-text PI-level eligibility.
- **Ranking = AI advisory, grounded** (`rank_and_explain`): Gemini re-ranks + gives a 1–2 sentence fit, each backed by a verbatim quote **verified against `synopsisDesc`** (unquotable quotes dropped, same whitespace-collapse pattern as `section_coach`). Falls back to API relevance order (no AI prose) when Gemini is down. `max_output_tokens=4096` (1536 truncated the ranking JSON → silent fallback). **The rank call passes `thinking_budget=0` + `timeout_s=8`, and `_MAX_RESULTS` is 8** (was 12): with 2.5-Flash "thinking" ON, the thinking tokens ate the 4096 output budget → the ranking JSON truncated → parse failed → the finder **silently fell back to unranked Grants.gov order with empty `fit_explanation`s**, AND the call took ~22s. Thinking OFF returns the full ranking in ~6s; the 8s timeout is a hard ceiling (falls back gracefully, never blows the <10s budget — Grants.gov itself is only ~1s).
- **Handoff:** "Start a proposal from this" routes the opp's solicitation URL into the existing `SolicitationUploadModal` via router state (`MyProposals` reads `location.state.solicitationUrl`; the modal gained an `initialUrl` prop), dropping the PI into the guided pathway. Tests: `tests/test_opportunity_finder.py` (unit) + `tests/test_opportunities_api_e2e.py` (route/auth).

## Caching (layers + lifetimes)
Caching exists to dodge the two real latency costs — the **Gemini call** and the **networked Cloud SQL read** from a **single-worker** backend — NOT to cache authoritative figures. **Cache the expensive (LLM answers), never the authoritative (budgets, statuses, verdicts, deadlines).**

**Chat answer cache** — `cache.py`, a 3-tier stack behind `query_cache.get/set` (keyed `md5(normalized_query : model)` — **no `user_id`**, so it's shared across users; personal/recall questions are gated out by `_should_cache`):
- **L1** in-memory (`cachetools.TTLCache`) — **24h**, 500 entries, per-instance.
- **L2** Redis — **7 days**, shared across instances, persistent. Degrades gracefully to "no L2" if Redis is down.
- **L3 semantic** (Google `text-embedding-004`, 256-dim) — matches re-worded questions at **cosine ≥ 0.95**; 100 in-memory entries + a 7-day Redis copy.
- **Citations** ride the same tiers under a `cit:` key (L1 24h / L2 7 days) so a cache HIT re-emits Sources.
- The chat HTTP responses are **`Cache-Control: no-cache`** (the cache is server-side only).

**Other server caches:** popular questions (`popular_questions.py`, in-process **6h** + Redis 7 days, daily-cron refreshed; the welcome-screen chips are now the **actual most-asked real user questions, Gemini-polished** — typos/grammar/semantic rewrite — and **PII-filtered** (`_is_personal_or_junk` drops SSN/salary/email/greetings/recall), NOT a fixed 36-question pool; `DEFAULT_QUESTION_POOL` is only the safe fallback/top-up. The one Gemini polish call runs on the daily refresh, never on a page load), Open Grants samples (`ogrants_finder.py`, **6h** TTLCache), forms catalog (`forms_catalog.py`, `@lru_cache` — **process-lifetime**, until redeploy). **Browser:** PWA service-worker precache of the app shell (workbox `autoUpdate` — until next deploy), Google-Fonts runtime `CacheFirst` (**1 year**), and `localStorage` (token/theme — until logout). TTLs are hard-coded constants per file (not yet env-configurable).

**Never cached:** personal-recall answers, errors/outages, transient "busy/try again" text, refusals (all gated in `cache.set`), and any answer's **personalized greeting** (stripped before storage — see Load-bearing fixes).

## Cron jobs (Cloud Scheduler → internal endpoints, auth via `X-Research-Secret` / `RESEARCH_SECRET`)
**6 endpoints — 5 recurring + 1 manual**: `/api/internal/research/run` (daily ~2am), `/api/internal/memory/consolidate` (daily ~3am — rolls conversations into long-term `UserMemory`), `/api/internal/memory/idle-sweep` (every 5 min), `/api/internal/deadlines/check` (daily — Deadline Watcher, idempotent so a retry never double-emails), `/api/internal/popular-questions/recompute` (daily — rebuilds the welcome-screen chips; this is the one Gemini polish call, kept off the page-load path), and `/api/internal/memory/backfill-profiles` (one-off). The actual schedules live in **Cloud Scheduler in GCP**, not the repo.

## Memory
Two tiers: **short-term** (current chat history) and **long-term** (`UserMemory` rows: department, role, active_grant, interests, …). The 3am consolidate cron moves chat facts into long-term; `mirror_profile_to_memories` mirrors saved profile fields into `UserMemory` on profile save. The chat endpoint also injects the user's **saved profile (name/department/title/role) directly from the `User` row every turn** as an authoritative `USER PROFILE` block — independent of memory selection — so "what department am I in?" always works (the top-N memory fetch alone could crowd the `[department]` row out). `build_memory_context` assembles long-term + semantic-recall + verbatim-turn sections for the agent.

## Load-bearing fixes (don't regress)
- **Gemini 429 on the chat path** (`vertex_agent.py`): a quota error arrives as response *text*; `_is_rate_limited` detects it, retries with backoff, then flags an outage — instead of laundering it into "system is busy" and amplifying via Pass-2 regeneration.
- **Two lanes — social chat (LANE 1) vs ORA (LANE 2)** (`agent.py` `BASE_INSTRUCTION` + `vertex_agent.py`): every chat message is either **LANE 1 social chat** (Gemini answers it itself, no KB) or **LANE 2 an ORA question** (answer ONLY from the KB). The `agent.py` instruction names both lanes explicitly; when a message could be either it must be treated as **LANE 2 and searched** — Gemini is FORBIDDEN to answer a possible ORA fact from its own training/memory (the strict "never fabricate a rate/deadline/name/ID" grounding preamble). **LANE 1** ("hi", "how are you", "thanks", "very good", "bye", plus bot-directed remarks like "do you miss me", "are you real?", "who are you") gets a warm KB-free reply — NOT the "I can only help with ORA" refusal, and NOT the "developed for Morgan State / ora.inavigator.ai" identity blurb (that blurb is only for "who made this app"). LANE 1 is answered by Gemini **naturally, to the message actually sent** — the per-kind example replies are now TONE guidance, not scripts. Do NOT re-add a fixed responder or force every pleasantry into one template (an earlier `smalltalk_reply()` fixed-per-category responder was removed as robotic; fix wording in the instruction instead). A substantive NON-ORA *knowledge* question (trivia, coding, recipes, other schools) is NOT LANE 1 — it is still refused per SECURITY. **Routing (`skip_kb_search`):** `_do_agent_pass` sets `state_delta["skip_kb_search"] = _is_smalltalk(msg) or _is_personal_identity(msg)`, and `agent.py`'s `before_model_callback` (`_select_model`) **drops the `VertexAiSearchTool` + KB prefetch** for those turns so the model physically can't run a stray KB search (which used to staple bogus Sources onto a greeting/identity turn). `_is_smalltalk` (whole message is pleasantry tokens via `_SMALLTALK_RE`, **OR** a bot-directed social question via `_SOCIAL_QUESTION_RE`) feeds BOTH this routing AND the Layer-3 verdict override (a LANE 1 turn is never graded "weak" and regenerated into a refusal) AND the `_is_non_kb_reply` Source-strip. Kept **NARROW**: `_is_personal_identity` (name/dept/role/email) and `_SOCIAL_QUESTION_RE` must NOT match institutional "my F&A rate" / "do you know the UEI?" — those stay LANE 2 and KEEP their KB search + Sources.
- **Evidence whitespace** (`section_coach.py` / `draft_critic.py`): `_verify_evidence` collapses whitespace before the substring match — see golden rule 2.
- **Grounded citations from the real KB search** (`vertex_agent.py` + `agent.py` + `kb_prefetch.py`): a *correct but uncited* chat answer was the symptom of the model answering from the **prefetched KB context** (injected into the instruction) instead of the native `VertexAiSearchTool` retrieval, so `groundingMetadata` came back empty (no Sources) AND it tripped the weak-grounding verdict (the slow Pass-2 regeneration). Two-part fix: **(B)** the prefetch block is now labeled **`[REFERENCE CONTEXT - FACT-CHECK ONLY]`** (was "use this to ground your answer") + grounding **rule 8** in `BASE_INSTRUCTION`, so the answer + Sources rest on the model's own KB search. **(C)** `_fallback_citations(query)` runs a live Vertex AI Search over the same unified datastore and resolves titles → URLs via `_get_kb_url_map()`, attached at the DELIVER step when a *trusted* ORA answer came back uncited — gated by `_wants_fallback_citations()` (skips small talk, refusals, outages, `kb_fail`, already-cited). Forcing the retrieval tool is **not** possible: `vertex_ai_search` is native retrieval grounding, and `tool_config mode=ANY` governs only `function_declarations`.
- **Chat latency budget** (`vertex_agent.py`): a turn can stack a full **Pass-2 regeneration** on Pass-1. `_over_latency_budget(elapsed)` (`CHAT_LATENCY_BUDGET_S`, code default 7s, **set to 5s in prod via `cloudbuild.yaml`**) skips the second agent round-trip when a *usable* Pass-1 answer graded weak and the turn has spent its budget — deliver Pass-1 + caution note instead (empty Pass-1 still regenerates; the streaming path already skips Pass-2). A `[LATENCY]` per-turn log (ms + verdict + chunks) makes the <10s target measurable. The bigger lever is warm instances (see Deploy) — cold starts, not the pipeline, were the ~40s.
- **Layer 3 grounding verification — surgical re-grounding REMOVED** (`vertex_agent.py` `_run_verified` / `_run_verified_stream`): the weak-answer path is now **grade (coverage ≥ 50% backed → "ok"; chunk count is NOT a gate) → LANE 1 / personal-recall guard (ship as-is) → latency bail (over budget → ship Pass-1 + `_WEAK_NOTE`) → strict-regenerate once (`_STRICT_PREFIX`, KB-only) → refuse (`_REFUSAL_MSG`)**. The streaming path can't regenerate, so a weak streamed answer just appends `_WEAK_NOTE`. The former **surgical per-sentence re-grounding** (`_unsupported_sentences` / `_surgical_reground` + grounded-span math + identifier-rejection, ~220 lines) was **deleted** — it fired **0 times in 148 production turns** (measured with `scripts/analyze_layer3_logs.py`, which parses the `[LAYER3]`/`[LATENCY]` logs). **Do NOT re-add it.** `_check_identifier_faithfulness` stays (still called by `_finalize_answer` for logging; the user-facing footer remains disabled), as do the coverage gate `_evaluate_grounding` and the latency bail.
- **Budget justification truncation** (`/api/budget/justification`): a non-empty-but-truncated AI fragment must fall back to the deterministic template (guard on the total figure), or the box shows a half-sentence.
- **Chat knows the profile** (`main.py` chat endpoint): inject the saved `User` profile every turn — see Memory section.
- **Cache poisoning** (`cache.py`): transient "busy / try again" messages must never be cached (they're not real answers).
- **Cached personalized greeting** (`cache.py`): the chat cache key has **no `user_id`** (answers are shared across users), so an answer that opens "Hello \<name>!" would replay one user's name to everyone who asks the same question. `_strip_personal_greeting` removes a leading salutation addressed to a proper name on **both write** (never persist a name) **and read** (sanitize entries poisoned before the fix). Kept narrow — a punctuation terminator is required, so generic openings ("Hello there, here's how…") and content ("Hello World Bank is a funder…") are left intact; the generic KB body stays cacheable.
- **Deploy:** `cloudbuild.yaml` must use the `ora-*` secret names and set the full Vertex env on both ADK and backend; the Cloud Build service account needs Cloud Run Admin + Service Account User. An **optional** secret (e.g. `ora-firecrawl-api-key`) must never hard-fail the deploy — the backend step self-heals a placeholder (see Secrets).
- **Stale PWA bundle → chunk 404 after deploy** (`frontend/nginx.conf`): returning PWA users hit 404s on lazy-loaded page chunks (e.g. the solicitation UI in `MyProposals-<hash>.js`) because a blanket `\.(js|css)$ → expires 1y; immutable` rule also cached **`index.html` and `sw.js`** — pinning the old service worker + shell so it kept requesting last build's hashed filenames. Fix: serve `index.html`, `sw.js`, `registerSW.js`, `manifest.webmanifest` as **no-cache/must-revalidate**; apply immutable-1y **only** to content-hashed files under `/assets/`. (Existing stale clients still need a one-time cache clear.)
- **No KB Sources on personal answers** (`vertex_agent.py`): a personal-recall question ("what department am I in?") is answered from the profile/chat history, not the KB — so it must NOT get KB citations stapled on. **Two paths, both closed:** (a) the FALLBACK path is gated by `_wants_fallback_citations` (skips small talk AND `_is_personal_recall`); (b) the model can ALSO run a stray KB search on a personal question and return REAL `groundingMetadata` citations, which bypass the fallback gate. Path (b) is closed by `_is_personal_identity` — a **narrow** name/department/role/email/"who am I" check, deliberately TIGHTER than `_is_personal_recall` (which also matches institutional "my F&A rate / my UEI / my required attachments" — those DO come from the KB and KEEP their Sources) — which clears `result["citations"]` in BOTH the `_run_verified` and `_run_verified_stream` DELIVER steps. (Disabling agent "thinking" for latency made the model call the search tool more often on personal questions, surfacing path (b).) **Update:** the two-lane routing (`skip_kb_search`, above) now **strips the search tool** for `_is_personal_identity` turns, so path (b) is prevented at source for those — the citation-clear is now belt-and-suspenders, and still covers `_is_personal_recall`'s broader institutional cases (which keep the tool + their legitimate Sources).
- **No KB Sources on off-topic refusals / pleasantries** (`vertex_agent.py`): same class as above but for **non-KB replies** — the **off-topic refusal** ("I can only help with Morgan State University Office of Research Administration questions."), greetings, outages, and **"thanks" acknowledgments** ("Great! I'm glad I could help."). The model runs a **stray KB search even on an off-topic / small-talk turn** (e.g. "who is the president of US?" or "thanks!") and returns REAL `groundingMetadata` citations, so the reply was rendered WITH a bogus `SOURCES: Staff: Farin Kamangar`. Fix: in BOTH the `_run_verified` and `_run_verified_stream` DELIVER steps, a single `_non_kb = _is_non_kb_reply(message, text)` guard **clears `result["citations"]` AND skips the `_wants_fallback_citations` fallback** (`if not _non_kb and …`) — clearing alone isn't enough because the fallback would immediately re-add Sources. `_is_non_kb_reply` fires when **(a)** the user's *message* is small talk (`_is_smalltalk` — covers praise/affirmations "very good"/"perfect"/"excellent"/"makes sense"/"good job" so `VERY GOOD.` classifies as small talk, **and bot-directed social questions** "do you miss me"/"are you real?"/"who are you" via `_SOCIAL_QUESTION_RE`), **(b)** the *reply* starts with a canned non-KB line (`_SKIP_GROUNDING_RE`: greetings / `I can only help with Morgan State` / outages / `You're welcome`), or **(c)** the *reply* is a **short** (≤240 char) conversational reply (`_NON_KB_REPLY_RE`: the greeting-response opener `I'm doing well/great/good`, `thanks for asking`, `glad/happy to help`, `no problem`, `my pleasure`, `you're welcome`). `_NON_KB_REPLY_RE` is deliberately NOT folded into `_SKIP_GROUNDING_RE` (which also gates grounding *verification* — a real answer opening "Great, the F&A rate is…" must not be exempted), its markers essentially never open a genuine grounded answer, and the length bound keeps a long real answer that says "happy to help you draft…" from losing its legitimate Sources. The `_SKIP_GROUNDING_RE`-only prefix match was insufficient: a greeting reply like `I'm doing well, thanks! How can I help you with ORA…` (to `VERY GOOD.`) matched none of those prefixes, so the stray Sources survived until the message/reply gates above were added. Verified: full backend suite green (682 passed).
- **ADK generate-config: "thinking" OFF + tighter token cap** (`adk_agent/…/agent.py`): the agent runs `generate_content_config` with `thinking_config=ThinkingConfig(thinking_budget=0)` and `max_output_tokens=2048` (was 4096). Gemini 2.5 Flash "thinking" is ON by default and adds seconds to every answer for a grounded retrieve+cite bot at temp 0.05 — the biggest single-pass latency win. `gemini_client` grew an **opt-in `thinking_budget` param** (default `None` = unchanged for every existing caller; only `opportunity_finder`'s rank call passes `0`). Restore 4096 / drop `thinking_config` if answers truncate or reasoning quality regresses.
- **Cache-hit citations** (`main.py` `/chat`, `/chat/stream`, `/chat/guest`): citations ride a parallel `cit:` cache key so a HIT re-emits Sources. **Known gap (not yet coded):** answers cached *before* Sources were stored (older popular questions like "what's the F&A rate?") replay with **empty** citations — a cache flush (admin `POST /api/admin/knowledge-base/sync-all` → `query_cache.clear()`) re-answers them fresh WITH Sources (a fresh answer stores `_fallback_citations`, verified working). A durable code fix would back-fill Sources on any cache hit missing them for a citable question.
- **Research agent = only real KB gaps, real answers** (`research_agent.py`): (1) `detect_and_log_failed_query` skips personal questions (`_is_personal_question`: personal-recall regex + PII + personal phrases) — a personal miss is NOT a KB gap. (2) `research_topic` uses the real Vertex project from `GOOGLE_CLOUD_PROJECT` (was a hardcoded, nonexistent `oranavigator-vertex-ai` → every research call 403'd → empty suggestions). Admin flow stays two-step: review → **Approve** → **Push to KB**.
- **Opportunity Finder** (`opportunity_finder.py`): `_is_open()` drops opportunities whose close date has passed (Grants.gov "posted" still returns expired/recurring ones); `fetch_opportunity` calls run **concurrently** (ThreadPoolExecutor) — the sequential fetches were the latency bottleneck. **Upcoming-only + rolling (product decision):** the open-date filter reads BOTH the synopsis `responseDate` AND the search2 `closeDate` (often the only one populated for standing programs) — a PAST date in EITHER excludes the opp, so an expired program with a blank `responseDate` no longer slips through. Rolling/no-deadline programs are KEPT and flagged `rolling: true` (UI labels them "Rolling — no fixed deadline"); results are stably partitioned **dated-upcoming first, rolling last** (the AI fit ranking still orders within each group).
- **Email verification bypass** (`SKIP_EMAIL_VERIFICATION=true` in `cloudbuild.yaml`, temporary): `/api/register` creates the verified user directly and skips the email link (no working sender wired up yet). Set to `false` / recreate the `ora-smtp` secret + re-add `SMTP_PASS=ora-smtp:latest` to restore verify-first. `email_service._send_email` no-ops when `SMTP_PASS` is empty.

## Known limitations / open work
- **NSF (and similar Akamai sites)** block server-side URL fetches by IP. **Firecrawl** (set `FIRECRAWL_API_KEY`, free tier ~500 one-time credits) is the reliable path; the Jina/allorigins fallbacks are throttled from Cloud Run. Locally the direct fetch works (residential IP isn't blocked). Uploaded PDFs always work regardless.
- Drafting-coach AI tailoring/feedback quality depends on the deployed Gemini being healthy; it degrades to deterministic checks otherwise.
- Demo aids live in repo root: `ORA_Navigator_Demo_Script.pdf`, `Sample_Solicitation_for_Demo.pdf` (generated by `scripts/make_demo_pdfs.py`).
- **Two copies of the KB metadata, and nothing syncs them.** The live **Vertex AI Search datastore** (`oranavigator-kb-v8`) is what the chatbot searches and what the admin dashboard edits. `backend/kb_structured/` is a **static snapshot committed to the repo**, read at import and cached for the process lifetime — it powers the **Forms catalog** (`forms_catalog.py` reads `_all_documents.jsonl`, 383 rows), `_get_kb_url_map()` (citation title→URL), and `proposal_templates.py`. So **editing a form document in the admin dashboard fixes the chatbot's answer but NOT the `/forms` page**, and a doc added to the datastore never appears under `/forms` — with no error either way. Refreshing the snapshot requires regenerating the JSONL and redeploying the backend.
