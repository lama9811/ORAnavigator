# CLAUDE.md

Canonical context for AI assistants working in this repo. If anything elsewhere disagrees, trust this file.

## What this is
**ORA Navigator** ‚Äî an AI assistant + proposal toolkit for Morgan State University's **Office of
Research Administration** (ORA): grants, compliance, pre/post-award, forms, ORA contacts, and a
guided workspace that helps inexperienced PIs actually write and submit fundable proposals.

## Three services
| Service | Stack | Local port | Container port |
|---|---|---|---|
| **frontend** | Vite + React 19 (PWA) | 3001 | 8080 |
| **backend** | FastAPI, **single-worker uvicorn** | 5002 | 5000 |
| **adk_agent** | Google ADK + Gemini 2.5 Flash (Vertex AI), `VertexAiSearchTool` over one unified KB datastore | 8081 | 8080 |

The chat path: frontend ‚Üí backend (`vertex_agent.py`) ‚Üí ADK agent (`adk_agent/ora_navigator_unified/`) ‚Üí Gemini on Vertex AI. The backend also serves the proposal toolkit (below) and talks to Cloud SQL (MySQL) + Redis.

## Build / test / deploy
- **Backend tests (keep green before any push/deploy):**
  ```bash
  cd backend && JWT_SECRET=test-secret TRUSTED_HOSTS=testserver,localhost,127.0.0.1 \
    python3 -m pytest -q --ignore=tests/test_agent_instruction.py
  ```
  Env vars are required because importing `main` constructs the app. `test_agent_instruction.py` needs `google-adk` (not always installed locally). App-level e2e tests use FastAPI `TestClient` with `dependency_overrides` for `get_db` / `get_current_user` (see `tests/test_proposals_api_e2e.py`).
- **Frontend:** `cd frontend && npm run build` (no `node_modules`? `npm install` first). It's a **PWA** ‚Äî verify changes in a fresh/incognito window.
- **Deploy:** pushes to **`main`** trigger **Cloud Build** (`cloudbuild.yaml`) which builds + deploys all three Cloud Run services in project `infra-vertex-494621-v1`, region `us-central1`. There's also a manual `deploy-cloudrun.sh` (needs local `gcloud`). The **ADK + backend** services run with **`--min-instances=1 --cpu-boost`** so the chat path never pays a 30‚Äì60s cold start (the dominant cause of slow first answers); the frontend stays `min-instances=0` (static, off the per-question path). **`cloudbuild.yaml` is a single point of failure** ‚Äî the backend deploy step is an inline bash heredoc, so a one-character typo there (e.g. a stray `=` swallowing `--min-instances`) breaks the *whole* deploy and silently blocks every merge from shipping. Validate it (`python3 -c "import yaml; yaml.safe_load(open('cloudbuild.yaml'))"`) after any edit.
- **Secrets** live in Secret Manager as `ora-database-url`, `ora-jwt-secret`, `ora-admin-email`, `ora-admin-password`, `ora-redis-url`, `ora-firecrawl-api-key` (mapped in `cloudbuild.yaml`). The Cloud Build runs as the **default compute service account** (`<projnum>-compute@‚Ä¶`), which lacks `secretmanager.secrets.create` ‚Äî so the `deploy-backend` step **self-heals** `ora-firecrawl-api-key` (creates a `unset` placeholder if missing) so a missing optional secret never breaks the deploy. Create real secret values via the Console (the project picker prevents the "wrong project" mistake).
- **`GOOGLE_GENAI_USE_VERTEXAI=TRUE`** must be set on the ADK **and** backend services (in `cloudbuild.yaml`) or Gemini falls back to the developer API and every call fails with "No API key was provided."

## Golden rules (followed everywhere; keep following them)
1. **AI is advisory; the deterministic core is authoritative.** Numbers, pass/fail, and eligibility verdicts are computed by code ‚Äî the LLM only explains, drafts prose, outlines, or gives advisory feedback. It can never change a figure.
2. **Ground every AI claim.** Any "covered/strong" finding must quote the source text; unquotable claims are dropped (`_verify_evidence` in `draft_critic.py` / `section_coach.py`). The membership check **collapses whitespace on both sides** (`" ".join(s.split())`) ‚Äî a pasted/hard-wrapped draft has newlines where Gemini's quote has spaces, and a raw substring match would wrongly reject a real quote and mark a complete section "NOT FOUND".
3. **Graceful fallback.** Every AI path returns a deterministic result when Gemini is unavailable (`gemini_client.generate_json/text` return `None` ‚Üí caller falls back). Never error because the model is down.
4. **Human reviews before commit.** Extractions and budgets are returned for review; nothing is saved until the user confirms.
5. **Conventions:** structured data stored as **TEXT columns + `json.dumps/loads`** (e.g. `Submission.budget_json`, `compliance_json`, `sections_json`, `notes`). New columns are added with a **self-healing migration** in `main.py:init_db()` (try `SELECT col`; on error `ALTER TABLE ADD COLUMN`) ‚Äî safe on prod MySQL, no Alembic.
6. **One feature = one focused change.** Don't refactor unrelated code. Verify before claiming done.
7. **Never push automatically.** Do NOT run `git push` (and don't open PRs) on your own ‚Äî only when the user explicitly asks in that message. Pushing to `main` triggers the Cloud Build deploy, so an unrequested push ships to production. Committing locally is fine to stage work, but wait for an explicit "push"/"deploy" before it leaves the machine. Approval to push once does not carry over to later changes.

## The Proposal Toolkit (backend `services/` + per-proposal modals in `frontend/src/components/`)
Each proposal (`Submission` + `SubmissionTask` in `models.py`, logic in `proposals_service.py`, sponsor task templates in `proposal_templates.py`) has a toolbar of tools:

- **Solicitation ingestion** ‚Äî `solicitation_extractor.py` (PDF‚Üítext via pdfplumber ‚Üí Gemini ‚Üí contract dict with `source_quotes`; fabricated quotes flagged via `_verify_source_quotes`). Two entry points: upload a PDF, or **paste a URL** (`url_fetcher.py`). The URL fetcher is SSRF-guarded (resolve + reject private/loopback/link-local IPs, pin the validated IP to defeat DNS rebinding). Some funder sites (NSF/Akamai) **block cloud-datacenter IPs**, so a direct fetch falls back through a scraper chain: **Firecrawl** (`_fetch_via_firecrawl`, preferred ‚Äî runs a real browser from non-blocked IPs; needs `FIRECRAWL_API_KEY`, and treats the `unset` deploy-placeholder as no-key), then `r.jina.ai` (set `JINA_API_KEY`), then `allorigins`. Uploaded PDFs are 100% local (pdfplumber) and never hit an IP block. `create_submission_from_solicitation` seeds the checklist; multi-category deadlines are surfaced via `deadline_details`. The extracted **eligibility text** is stored in `notes` and fed to the Drafting Coach.
- **Budget Helper** ‚Äî `budget_helper.py`: deterministic F&A/MTDC/fringe math (Morgan's real rates), sponsor-cap check. **Coaching layer:** `budget_advisories()` (sanity flags), `suggest_trims()` (concrete cuts to get under cap), category/F&A/fringe tooltips. **v2:** multi-year projection with escalation (`compute_budget` detects `project_years`; single-year output unchanged), CSV export (`budget_to_csv` + `/budget.csv`), per-line justification, and a **deterministic prose** budget justification (`draft_justification` ‚Äî labeled paragraphs, reads like a real submission). AI only polishes the prose and is forbidden to change figures; the `/api/budget/justification` endpoint falls back to the template when the AI output is empty **or truncated** (missing the total-cost figure ‚Äî a Gemini-under-load failure mode). The UI **effort‚Üíperson-months helper is read-only**, derived from each person's `effort_pct` in the People rows (not a separate input), so it can never disagree with the budget. **Per-category caps:** when a solicitation defines multiple proposal categories with different award maxima (e.g. NSF IDSS Category I/II/III), `solicitation_extractor` returns `budget_cap_details` (`[{category, cap}]`, a stated **range ‚Üí max**), persisted as a `Category caps:` line in `notes` (round-tripped by `reconstruct_solicitation_context`; category names sanitized of `;`/em-dash); the single `budget_cap` stays the smallest. The modal shows a **Funding category** picker that fills the Sponsor cap ‚Äî default is a blank "Select your category‚Ä¶" so the PI consciously picks; the picker appears only for 2+ categories. **Spreadsheet view:** `compute_budget` also returns a render-ready `table` (`{columns, rows}`; single-year = one `Amount` column, multi-year = a column per year + `Total`) built by `_budget_table` from the per-year breakdowns (now retained instead of discarded). The modal renders it as a read-only **Budget table** grid ‚Äî it formats numbers but computes none, so it always matches the summary and the CSV (row order mirrors `budget_to_csv`). **Modal layout:** the Budget table's `<table>` uses class **`bh-table`** ‚Äî NOT `bh-grid`, which is the form's two-column `display:grid` and silently turned the table into a broken 2-column grid (thead/tbody split, amounts clipped). The whole modal body (form + summary + table + justification) scrolls as **one** `.bh-scroll` region; the navy **Summary is `position:sticky`** so the total stays in view while the taller form scrolls (disabled on mobile). The **justification draft panel has a Close (√ó)** that clears `justification`+`perLine` without closing the whole modal. **v3 (multi-year coaching):** the multi-year path is otherwise unchanged but now *explains itself* to a first-time PI ‚Äî `ESCALATION_GUIDANCE` tooltip (escalation hits **salaries only**, ~2-3% federal norm; exposed via `rate_options()`), per-year **salary deltas** in `multi_year.years` (`salary_delta`/`salary_delta_pct`, shown as "salary +$X (Y%)"), the cap badge spells out it's the **cumulative N-year TOTAL** (`multi_year.cap_basis="cumulative"`), `draft_justification` adds a **"Period of performance"** paragraph (figures from `multi_year`; placed *before* the final total so the endpoint truncation guard still trips), and `multi_year_advisories()` flags **0% escalation** (info) / **>10%** (warn) ‚Äî appended to `advisories` at the `compute_budget` level since `_compute_single` never sees `escalation_pct`. All additive/deterministic; AI still only polishes. Out of scope (deliberately): per-person escalation rates and ramp-up/down spending profiles.
- **Drafting Coach** ‚Äî `section_coach.py`: per-section **outline** + advisory **feedback** on the PI's own draft (covered/missing elements, grounded). Coaching only ‚Äî never writes prose. Section menu is **by sponsor** (`_SPONSOR_ORDER`: NSF ‚Üí Project Summary / Project Description / Broader Impacts / Data Management Plan; NIH ‚Üí Specific Aims / Research Strategy / DMP; else generic). **v2:** save drafts (`sections_json`), live length meter (`WORD_TARGETS`), "match THIS solicitation" (feeds required attachments/eligibility/page limits), deterministic `clarity_check()`. **v3 (solicitation-aware):** `outline_section` now USES the solicitation context it already received ‚Äî outline tips reflect this funder's page limits/required attachments/eligibility, tailoring runs on **topic OR context** (not just topic), and the "match THIS solicitation" panel shows in **outline** mode too; `SECTION_SAMPLES` maps each section to an authored **Sample Proposals Library** entry (validated via `get_sample` so a link can't 404), surfaced as a "See a worked example" link in both modes; `review_section` (via `_REVIEW_SYSTEM`) also flags solicitation requirements the draft doesn't address (evidence-grounding unchanged). **v4 (reviewer-lens + cross-section coherence):** **(A)** a deterministic per-sponsor `REVIEW_RUBRICS` / `review_rubric(sponsor)` ‚Äî the panel's actual scoring criteria (NSF Intellectual Merit + Broader Impacts; NIH Significance/Innovation/Approach/Investigators/Environment; generic fallback) ‚Äî is attached to **both** outline and review results (so the PI writes TO the rubric), and `review_section` asks for advisory `reviewer_notes` (one per criterion, "how a panel would judge THIS draft"). Notes are **opinions/questions, NOT coverage claims** (no quote required); coverage stays in the grounded `checklist`. Notes are filtered to valid criteria; empty when AI is off (rubric always present). **(B)** `coherence_check(sponsor, drafts, context, budget)` ‚Äî a NEW advisory check that compares the PI's *saved* sections against each other: NIH Specific Aims ‚Üî Research Strategy, scope ‚Üî eligibility, timeline ‚Üî budget (`_coherence_candidate_pairs` builds only the pairs whose source sections exist). Every `"aligned"` verdict must quote **both** sides, verified with the shared `_quote_in()` helper (the whitespace-collapse match, **refactored out of `_verify_evidence`** ‚Äî golden rule 2); unverifiable ‚Üí demoted to `"unclear"`, quote dropped. Deterministic fallback when AI is off (incl. an Aim 1/2/3 label co-occurrence pass; never fabricates quotes). `ready:false` when <2 sections saved or no pair applies. Endpoint **`POST /api/me/submissions/{id}/coherence`** (reads existing `sections_json`/`budget_json`/`notes` ‚Äî no new state); UI is a third **"Cross-section"** tab gated on ‚â•2 saved drafts, plus a `ReviewerPanel` rendered in outline+review. **v5 (solicitation responsiveness matrix):** `responsiveness_matrix(sponsor, drafts, context)` ‚Äî a NEW whole-proposal advisory check that maps each explicit solicitation **ask** onto the PI's *saved* drafts, attacking the #1 triage reason (non-responsiveness). The requirements list is assembled **deterministically** by `_responsiveness_requirements` from already-grounded state ‚Äî page-limited named sections (`context["page_limits"]` keys ‚Üí `SECTIONS` labels), required attachments/elements (`context["required_attachments"]`), the sponsor's `review_rubric()` criteria, and eligibility text ‚Äî **so the AI can never invent an ask** (it receives a fixed `id` set and only assigns coverage). Status vocabulary is `addressed | partial | not_found | check_by_hand` (draft_critic's `missing|unclear` renamed) ‚Äî **coverage only, never a score/verdict** (respects the removed-Fundability boundary below). Every `addressed`/`partial` must quote a saved draft verbatim, verified with `_quote_in`; unverifiable ‚Üí demoted to `check_by_hand`, quote dropped, and `where` is re-attributed authoritatively to the section containing the verified quote (not trusted from the model). Deterministic fallback marks every row `check_by_hand` (never guesses `not_found` ‚Äî offline it can't prove absence ‚Äî and never fabricates a quote). Dedupe by normalized label (DMP as both attachment + page-limited section collapses to one, section wins). Endpoint **`POST /api/me/submissions/{id}/responsiveness`** (reads `sections_json`/`notes` ‚Äî no new state, no budget); UI is a fourth **"Responsiveness"** tab gated on ‚â•1 saved draft. **UI removed (2026-07):** the **Cross-section** (v4 coherence) and **Responsiveness** (v5) tabs were removed from the Drafting Coach UI by product decision ‚Äî the Coach now offers only "Outline it" and "Feedback on my draft". Both **backend endpoints (`/coherence`, `/responsiveness`) and services remain** (now unused by the UI); re-add the tabs to restore them.
- **Draft Critic vs Drafting Coach** ‚Äî different jobs: the **Coach** grades the *writing* of one section; the **Critic** (`draft_critic.py`) checks the *whole package's completeness/compliance* (required sections & **solicitation-required attachments**, page limits, budget vs cap) + advisory AI review. Completeness is the Critic + the solicitation checklist, never the Coach.
- **Guided steps** ‚Äî `task_guidance.py`: keyword-matched how-to + sample per checklist task, attached in `_submission_task_to_dict`; per-task "add to calendar".
- **Guided Proposal Pathway** (the proposal detail view in `MyProposals.jsx`, presentation/orchestration ‚Äî **no new tool logic**) ‚Äî for inexperienced PIs, the flat toolbar is a **staged path**: Set up ‚Üí Build ‚Üí Review & submit, each tool carrying a ‚úì status badge from serializer flags (`has_budget`/`has_compliance`/`has_sections` = `bool(sections_json)`; Solicitation from the `hasSolicitation` notes-regex; Critique is stateless). A pure `nextStep(submission)` picks the single recommended action ‚Äî only that button gets the accent (the rest share one calm style) ‚Äî and a plain-language **"What to do next"** card explains the current step + why (active proposals only). Delete + Add-to-Calendar live in a `‚ãØ` overflow. The summary shows the **internal ORA routing deadline** (`proposals_service.internal_routing_deadline` ‚Üí `internal_deadline` field, **5 business days before** the sponsor date, weekends skipped) so a first-timer plans backward from the real institutional cutoff.
- **Draft Critic** (`draft_critic.py`) ‚Äî mechanical pre-submission checks (page limits, required sections/attachments, budget vs cap) + advisory AI review.
- **Compliance Sentinel** (`compliance_sentinel.py`) ‚Äî IRB/IACUC/COI/RCR/export rules. **Deadline Watcher** (`deadline_watcher.py`) ‚Äî AI-personalized reminder emails (cron). `ics_export.py` ‚Äî calendar feeds.

**Removed:** the former **Fundability / Eligibility self-check** tool (reviewer-lens scoring + go/no-go) was deleted per product decision ‚Äî `services/fundability.py`, `services/eligibility.py`, their tests, the `FundabilityModal`, and the `/fundability/criteria` ¬∑ `/eligibility` ¬∑ `/fundability` endpoints are gone. Do not re-add. The eligibility *text* extracted from the solicitation is still kept (`_eligibility_text_from_notes`) and fed to the Drafting Coach.

## Sample Proposals Library (standalone page, not a per-proposal tool)
A browse surface at **`/sample-proposals`** (navbar "Samples" button ‚Üí `frontend/src/components/SampleProposalsLibrary.jsx`, styled like `FormsCatalog`) so inexperienced PIs can see what a fundable proposal actually reads like. It's a **hybrid** of three sources, ranked authored-first and filtered by category chip (NSF / NIH / Foundations / Early-career):
- **type `"pdf"`** ‚Äî **original** sample proposals we authored ourselves and host for download (zero copyright exposure; each PDF carries a "not a real submitted proposal" disclaimer). Three today: a full NSF planning proposal (EJ-IDSS), an NIH Specific Aims + Research Strategy excerpt, and an annotated Budget Justification.
- **type `"link"` (static)** ‚Äî vetted links OUT to authoritative public libraries (NSF's official sample, NIH/NIAID funded apps + reviewer critiques, USF/UAF/Northwestern/UChicago/SERC, Open Grants, Morgan ORA). We **never rehost** these ‚Äî they're copyrighted and often index pages, not single files.
- **type `"link"` (live, `community:true`)** ‚Äî **Open Grants** (ogrants.org) researcher-shared proposals, fetched LIVE in `services/ogrants_finder.py` (mirrors the Grants.gov boundary style: fixed host `api.github.com`, graceful `[]` on failure, never fabricates). One cached GitHub **tarball** call (`TTLCache`, 6h TTL ‚Üí new ogrants entries appear within ~6h; serves last-good on failure) ‚Üí parse each `_grants/*.md` flat front-matter (`_parse_front_matter`, no PyYAML) ‚Üí `_normalize` to the card shape (**always `link`** ‚Äî the repo holds only metadata + an external link, NO full proposal text; entries with no title or https link are skipped). `_categories_for` maps funder/program onto the chips. Merged authored-first in the `GET /api/sample-proposals` endpoint (the static `sample_proposals.py` list stays untouched). UI shows a **"Community"** badge.

**Interest ranking:** a PI can type their research in an Opportunity-Finder-style box and the whole shelf reorders best-first. `rank_samples(items, query)` is **deterministic keyword overlap** (no LLM): title matches weigh 3√ó, **whole-word** matching (so "AI" ‚âÝ "tr**ai**ning"; small `_KEEP_SHORT` allowlist keeps AI/ML/‚Ä¶ despite <3 chars), unmatched items keep authored-first order and sort last, each match carries `{score, terms}`. Endpoint **`POST /api/sample-proposals/search`** (auth'd ‚Äî the only sample endpoint that is, so it can fold in the PI's **saved `UserMemory` interests** like the Opportunity Finder); chips still filter in place; an empty query / zero matches reverts to browse order and the UI says so honestly ("no close matches yet ‚Äî showing the full library").

Data + helpers for the authored/static entries are a constant in `services/sample_proposals.py` (mirrors `forms_catalog.py` / `DEFAULT_QUESTION_POOL` ‚Äî no DB, no LLM; the only network is the live ogrants merge, isolated in `ogrants_finder.py`). Endpoints: public `GET /api/sample-proposals?category=` (listing) and public `GET /api/sample-proposals/{id}/download` (FileResponse; 404 for unknown/link/missing), plus the **auth'd** `POST /api/sample-proposals/search` (interest ranking + saved interests). The page sits behind `RequireAuth` like `/forms`; the two GETs are auth-free (the content is all public), the search POST is not. **Authored PDFs** are generated by **`scripts/make_sample_proposals.py`** (reportlab, dev-only) into **`backend/sample_proposals/`** and are **committed** so they ship in the Docker image ‚Äî `.gitignore` has a `!backend/sample_proposals/*.pdf` negation past the blanket `*.pdf` rule (without it the download 404s in prod). The backend needs **no runtime PDF dep** ‚Äî it only serves the pre-generated files. To add/edit a sample: change the prose in the script, re-run it, add an entry to `SAMPLE_PROPOSALS`, commit the new PDF. (The root one-offs `scripts/make_demo_pdfs.py` / `make_nsf_real_draft.py` are intentionally **local-only**, not in the repo.)

## Opportunity Finder + Fit Advisor (standalone page, not a per-proposal tool)
A discovery surface at **`/opportunities`** (navbar "Find Funding" button ‚Üí `frontend/src/components/OpportunityFinder.jsx`, styled like the Samples/Forms surfaces) that closes the gap *before* a PI has a solicitation: they describe their research in plain text and get a ranked list of **live, open federal opportunities**, each with a grounded fit explanation, an eligibility verdict, and a mechanism note. Logic in **`services/opportunity_finder.py`**; endpoint **`POST /api/opportunities/search`** (auth'd; enriches the query with the user's saved interests from `UserMemory`).
- **Source = Grants.gov public REST API** (fixed host `api.grants.gov`, no key, covers all 26 federal agencies ‚Äî NSF/NIH/DOE/‚Ä¶): `search2` (keyword ‚Üí opp ids/basics) then `fetchOpportunity` per top result (the rich synopsis: `applicantTypes`, `applicantEligibilityDesc`, `synopsisDesc`, award ceiling, close date, contact). Foundations are out of v1 (no free API). Returns `[]` (never fabricates) on API failure.
- **Eligibility = deterministic STOP-gate** (`eligibility_gate`): Morgan State's baked-in profile (public IHE / HBCU / MSI ‚Äî like the F&A rates are baked in) matched against the opportunity's applicant-type *descriptions* ‚Üí `eligible` / `unrestricted` / `see_text` / `ineligible`. AI advisory only for the free-text PI-level eligibility.
- **Ranking = AI advisory, grounded** (`rank_and_explain`): Gemini re-ranks + gives a 1‚Äì2 sentence fit, each backed by a verbatim quote **verified against `synopsisDesc`** (unquotable quotes dropped, same whitespace-collapse pattern as `section_coach`). Falls back to API relevance order (no AI prose) when Gemini is down. `max_output_tokens=4096` (1536 truncated the ranking JSON ‚Üí silent fallback).
- **Handoff:** "Start a proposal from this" routes the opp's solicitation URL into the existing `SolicitationUploadModal` via router state (`MyProposals` reads `location.state.solicitationUrl`; the modal gained an `initialUrl` prop), dropping the PI into the guided pathway. Tests: `tests/test_opportunity_finder.py` (unit) + `tests/test_opportunities_api_e2e.py` (route/auth).

## Caching (layers + lifetimes)
Caching exists to dodge the two real latency costs ‚Äî the **Gemini call** and the **networked Cloud SQL read** from a **single-worker** backend ‚Äî NOT to cache authoritative figures. **Cache the expensive (LLM answers), never the authoritative (budgets, statuses, verdicts, deadlines).**

**Chat answer cache** ‚Äî `cache.py`, a 3-tier stack behind `query_cache.get/set` (keyed `md5(normalized_query : model)` ‚Äî **no `user_id`**, so it's shared across users; personal/recall questions are gated out by `_should_cache`):
- **L1** in-memory (`cachetools.TTLCache`) ‚Äî **24h**, 500 entries, per-instance.
- **L2** Redis ‚Äî **7 days**, shared across instances, persistent. Degrades gracefully to "no L2" if Redis is down.
- **L3 semantic** (Google `text-embedding-004`, 256-dim) ‚Äî matches re-worded questions at **cosine ‚â• 0.95**; 100 in-memory entries + a 7-day Redis copy.
- **Citations** ride the same tiers under a `cit:` key (L1 24h / L2 7 days) so a cache HIT re-emits Sources.
- The chat HTTP responses are **`Cache-Control: no-cache`** (the cache is server-side only).

**Other server caches:** popular questions (`popular_questions.py`, in-process **6h** + Redis 7 days, daily-cron refreshed), Open Grants samples (`ogrants_finder.py`, **6h** TTLCache), forms catalog (`forms_catalog.py`, `@lru_cache` ‚Äî **process-lifetime**, until redeploy). **Browser:** PWA service-worker precache of the app shell (workbox `autoUpdate` ‚Äî until next deploy), Google-Fonts runtime `CacheFirst` (**1 year**), and `localStorage` (token/theme ‚Äî until logout). TTLs are hard-coded constants per file (not yet env-configurable).

**Never cached:** personal-recall answers, errors/outages, transient "busy/try again" text, refusals (all gated in `cache.set`), and any answer's **personalized greeting** (stripped before storage ‚Äî see Load-bearing fixes).

## Cron jobs (Cloud Scheduler ‚Üí internal endpoints, auth via `X-Research-Secret` / `RESEARCH_SECRET`)
4 recurring + 1 manual: `/api/internal/research/run` (daily ~2am), `/api/internal/memory/consolidate` (daily ~3am ‚Äî rolls conversations into long-term `UserMemory`), `/api/internal/memory/idle-sweep` (every 5 min), `/api/internal/deadlines/check` (daily ‚Äî Deadline Watcher, idempotent so a retry never double-emails), and `/api/internal/memory/backfill-profiles` (one-off). The actual schedules live in **Cloud Scheduler in GCP**, not the repo.

## Memory
Two tiers: **short-term** (current chat history) and **long-term** (`UserMemory` rows: department, role, active_grant, interests, ‚Ä¶). The 3am consolidate cron moves chat facts into long-term; `mirror_profile_to_memories` mirrors saved profile fields into `UserMemory` on profile save. The chat endpoint also injects the user's **saved profile (name/department/title/role) directly from the `User` row every turn** as an authoritative `USER PROFILE` block ‚Äî independent of memory selection ‚Äî so "what department am I in?" always works (the top-N memory fetch alone could crowd the `[department]` row out). `build_memory_context` assembles long-term + semantic-recall + verbatim-turn sections for the agent.

## Load-bearing fixes (don't regress)
- **Gemini 429 on the chat path** (`vertex_agent.py`): a quota error arrives as response *text*; `_is_rate_limited` detects it, retries with backoff, then flags an outage ‚Äî instead of laundering it into "system is busy" and amplifying via Pass-2 regeneration.
- **Greetings / small talk** (`vertex_agent.py` `_is_smalltalk` + `agent.py` GREETINGS rule): "hi", "how are you", "thanks" get a warm KB-free reply ‚Äî NOT the "I can only help with ORA" refusal, and NOT the "developed for Morgan State / ora.inavigator.ai" identity blurb (that blurb is only for "who made this app"). The grounding gate must NOT grade a greeting reply "weak" and regenerate it (that turned it back into a refusal).
- **Evidence whitespace** (`section_coach.py` / `draft_critic.py`): `_verify_evidence` collapses whitespace before the substring match ‚Äî see golden rule 2.
- **Grounded citations from the real KB search** (`vertex_agent.py` + `agent.py` + `kb_prefetch.py`): a *correct but uncited* chat answer was the symptom of the model answering from the **prefetched KB context** (injected into the instruction) instead of the native `VertexAiSearchTool` retrieval, so `groundingMetadata` came back empty (no Sources) AND it tripped the weak-grounding verdict (the slow Pass-2 regeneration). Two-part fix: **(B)** the prefetch block is now labeled **`[REFERENCE CONTEXT - FACT-CHECK ONLY]`** (was "use this to ground your answer") + grounding **rule 8** in `BASE_INSTRUCTION`, so the answer + Sources rest on the model's own KB search. **(C)** `_fallback_citations(query)` runs a live Vertex AI Search over the same unified datastore and resolves titles ‚Üí URLs via `_get_kb_url_map()`, attached at the DELIVER step when a *trusted* ORA answer came back uncited ‚Äî gated by `_wants_fallback_citations()` (skips small talk, refusals, outages, `kb_fail`, already-cited). Forcing the retrieval tool is **not** possible: `vertex_ai_search` is native retrieval grounding, and `tool_config mode=ANY` governs only `function_declarations`.
- **Chat latency budget** (`vertex_agent.py`): a turn can stack a full **Pass-2 regeneration** on Pass-1. `_over_latency_budget(elapsed)` (`CHAT_LATENCY_BUDGET_S`, default 7s) skips the second agent round-trip when a *usable* Pass-1 answer graded weak and the turn has spent its budget ‚Äî deliver Pass-1 + caution note instead (empty Pass-1 still regenerates; the streaming path already skips Pass-2). A `[LATENCY]` per-turn log (ms + verdict + chunks) makes the <10s target measurable. The bigger lever is warm instances (see Deploy) ‚Äî cold starts, not the pipeline, were the ~40s.
- **Budget justification truncation** (`/api/budget/justification`): a non-empty-but-truncated AI fragment must fall back to the deterministic template (guard on the total figure), or the box shows a half-sentence.
- **Chat knows the profile** (`main.py` chat endpoint): inject the saved `User` profile every turn ‚Äî see Memory section.
- **Cache poisoning** (`cache.py`): transient "busy / try again" messages must never be cached (they're not real answers).
- **Cached personalized greeting** (`cache.py`): the chat cache key has **no `user_id`** (answers are shared across users), so an answer that opens "Hello \<name>!" would replay one user's name to everyone who asks the same question. `_strip_personal_greeting` removes a leading salutation addressed to a proper name on **both write** (never persist a name) **and read** (sanitize entries poisoned before the fix). Kept narrow ‚Äî a punctuation terminator is required, so generic openings ("Hello there, here's how‚Ä¶") and content ("Hello World Bank is a funder‚Ä¶") are left intact; the generic KB body stays cacheable.
- **Deploy:** `cloudbuild.yaml` must use the `ora-*` secret names and set the full Vertex env on both ADK and backend; the Cloud Build service account needs Cloud Run Admin + Service Account User. An **optional** secret (e.g. `ora-firecrawl-api-key`) must never hard-fail the deploy ‚Äî the backend step self-heals a placeholder (see Secrets).
