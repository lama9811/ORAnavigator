ORA Navigator

An AI Research-Administration Assistant for Morgan State University

Project Pitch · Office of Research Administration (ORA) · Live at ora.inavigator.ai

What it is. ORA Navigator is a full-stack AI assistant for Morgan State faculty, principal investigators, research staff, and department administrators. They ask questions about grants, compliance, forms, and post-award management in plain English and get answers grounded in 382 official ORA documents — every answer cites its morgan.edu source. Beyond chat, it provides a suite of tools to actually run a grant proposal from idea to award.

2How an Answer Is Produced — the RAG Pipeline

The core engineering challenge is trust: the AI may only answer from Morgan's documents, never guess. This is enforced at three layers, not just by asking the model nicely.

Your question → quick checks (greeting? asked before → cache? "list X" → instant menu)

Layer 1 — head start: a fast keyword (TF-IDF) prefetch grabs the top-5 likely docs and injects them into the AI's instructions (~30 ms, no AI cost).

Layer 2 — the real retriever: the agent calls Vertex AI Search as a tool to find passages by meaning, drafts an answer, and the matching morgan.edu URLs become the "Sources" block.

Layer 3 — the fact-checker (the guarantee): a grounding gate checks the result. If it's weak, it forces a strict "KB-only" rewrite; if still nothing, it refuses and routes the user to ORA. It doesn't trust the model's confidence — it verifies the output.

Save to cache → send the answer + Sources.

4Memory — Personalization Across Sessions

The assistant remembers a user's research-admin context (role, department, active grant, IRB protocol, …) so answers get more personal over time:

  • Rolling session summary + embedding-based semantic recall over a user_memories table — it surfaces relevant past facts even across different conversations.
  • Real-time extraction every few turns, plus an idle-sweep (every 5 min) so mid-session facts aren't lost.
  • A nightly 3 a.m. consolidation job distills durable one-line facts from the day's chats (PII/financial details excluded), de-duplicates, and caps at 5 memories per type. It reads chat history only — it never deletes it.
  • Profile fields (Department / Title / Role / Interests) are mirrored into memory, so they feed the agents with no extra code.

6Caching & Knowledge Base

3-layer cache (speed + cost)

  • L1 — in-memory per instance (24 h)
  • L2 — Redis, shared (7 days)
  • L3 semantic — embeds the question; a 0.95-cosine match returns a prior answer even if reworded

A cache hit skips the LLM entirely — instant, free.

Knowledge base

  • 382 documents scraped from the live ORA website, kept in a hierarchical tree (compliance, pre/post-award, forms, trainings…).
  • Indexed into a Vertex AI Search datastore the agent queries at runtime.
  • Mirrors the real site nav, so answers map to pages users can actually find.

7Data, Auth & Engineering Quality

  • Database (Cloud SQL MySQL 8.4): users, chat history, proposals & tasks, memories, suggested questions, feedback, support tickets, deadline-reminder log.
  • Auth: sign-up restricted to @morgan.edu with email verification; JWT sessions + bcrypt; user / admin roles with an admin dashboard.
  • Trust by design: anything that must be exact (budget math, compliance) is produced by code, not the LLM; the grounding gate + KB-only regeneration prevent fabricated facts.
  • Quality: 437 automated backend tests, a 108-case faithfulness/hallucination evaluation harness, and a safe CI/CD deploy pipeline to Cloud Run.
The headline idea to pitch: ORA Navigator isn't "a chatbot." It's a trustworthy research-administration platform where the AI is deliberately kept on a short leash — it answers only from Morgan's own documents and never decides money or compliance on its own. That design is what makes it safe for faculty to actually rely on.
