What ORA Navigator Is

Page 1 of 5

An AI assistant and proposal toolkit for Morgan State University's Office of Research Administration — answering grant, compliance and pre/post-award questions only from the University's own documents.

The one-sentence version

ORA Navigator is a search engine over Morgan State's own research-administration documents, with an AI that writes the answer in sentences. It cannot browse the web, and it is built to refuse rather than guess. Two consequences follow from that design: every answer can be traced to a document, and the system is only ever as good as the library the ORA staff maintain.

Who it serves, and the problem it solves

An inexperienced Principal Investigator (PI) faces two separate problems: finding out the rules, and actually producing a fundable proposal. ORA Navigator covers both — a grounded chat assistant for the first, and a guided proposal workspace for the second (budgets, drafting coaching, compliance and pre-submission checks).

The three services

The system is three independently deployed programs, so a failure stays in its lane. When Google's model was rate-limited, the proposal toolkit kept working — only chat degraded.

Where it runs

All three deploy to Google Cloud Run in project infra-vertex-494621-v1 (region us-central1). A push to main triggers Cloud Build, which builds and deploys all three. Persistent data lives in Cloud SQL (MySQL); the shared cache is Redis; the document library is a Vertex AI Search datastore.

A load-bearing constraint: the ADK service must stay a single instance (--min-instances=1 --max-instances=1). It holds agent sessions in memory, so with two instances a session created on one returns 404 on the other and the user sees an intermittent error. Never scale it horizontally without first configuring a shared session store (ADK_SESSION_DB_URI).

The four governing rules

  1. AI is advisory; the deterministic core is authoritative. Budget totals, pass/fail and eligibility verdicts are computed by ordinary code. The model may explain a figure; it may never change one.
  2. Ground every claim. Any "covered" finding must quote its source verbatim; unquotable claims are dropped.
  3. Degrade, never fail. Every AI path has a deterministic fallback for when the model is unavailable.
  4. A human reviews before anything is committed. Extractions, budgets and researched KB additions are proposed, never auto-published.

The Stack: Front End and Back End

What each layer is built from, what it owns, and where the data actually lives.

Front end

Vite + React 19, shipped as an installable PWA (Progressive Web App): it precaches its own shell with a Workbox service worker, so it loads instantly and can be added to a phone home screen. Routing is client-side; authenticated pages sit behind a RequireAuth wrapper.

  • Chat with streaming answers, sources, and chat history in a sidebar.
  • My Proposals — the guided pathway and every per-proposal tool modal.
  • Find Funding, Samples, Forms — three browse surfaces.
  • Admin dashboard — seven tabs: Home, Database, Tickets, Feedback, Users, Research, System.
PWA caching caveat: index.html, sw.js and the manifest are served no-cache; only content-hashed files under /assets/ get the immutable one-year policy. A blanket rule once pinned the old service worker and caused chunk 404s for returning users after a deploy.

Back end

FastAPI on a deliberately single-worker uvicorn. It is the only component that talks to the database, and it owns every rule that must not be negotiable:

  • Auth — JWT, with registration gated to @morgan.edu addresses server-side.
  • The proposal toolkit — solicitation ingestion, budget maths, drafting coach, draft critic, compliance sentinel (all in backend/services/).
  • The safety layer around chat: grounding verification, citation handling, caching.

A convention worth knowing

Structured data is stored as TEXT columns holding JSON (budget_json, sections_json, compliance_json), and new columns are added by a self-healing migration at startup — try SELECT the column, and on error ALTER TABLE ADD COLUMN. There is no Alembic; this is safe against production MySQL.

Two copies, no sync. The Forms catalog reads the bundled kb_structured/_all_documents.jsonl snapshot, not the live datastore. Editing a form document in the admin dashboard changes the chatbot's answer but not the Forms page, and nothing warns you the two have drifted.

The ADK Agent and the RAG Pipeline

Page 3 of 5

How one question becomes one grounded, cited answer — and the layers that stop the model inventing facts.

What ADK is doing

ADK is Google's Agent Development Kit. The adk_agent service hosts a single agent, ora_navigator_unified, which pairs Gemini 2.5 Flash with a VertexAiSearchTool pointed at one unified datastore. The backend calls it over /run_sse and streams the result back.

Its generation settings are tuned for a grounded retrieve-and-cite bot rather than a creative one: temperature 0.05, max_output_tokens 2048, and thinking disabled (thinking_budget=0) — the single biggest latency win, since 2.5 Flash reasons by default and that time is wasted on lookup-and-quote work.

RAG — retrieval before generation

A general chatbot answers from what it absorbed in training. This one is forbidden to. The retrieval step runs first, and the model is handed the passages:

  1. Route the message. Is this small talk, a personal question, or an ORA question?
  2. Retrieve. The agent searches the datastore. Matching is by meaning, not keywords — "who do I call about human subjects?" finds the IRB contact without the word "IRB".
  3. Generate. Gemini writes an answer using only those passages.
  4. Verify. The answer is graded on how much of it is actually supported.
  5. Deliver — with a Sources line, or regenerate, or refuse.

Two lanes

Lane 1 — social chat

"hi", "thanks", "how are you", "who are you". Answered directly and warmly. The search tool is removed from the model entirely for these turns, so it physically cannot staple bogus sources onto a greeting.

Lane 2 — an ORA question

Searched, grounded, cited. When a message could be either, it is treated as Lane 2 and searched — erring toward retrieval is safer than answering from memory. A non-ORA knowledge question is refused, not chatted.

The grounding layers — three, in order

#LayerWhenWhat it does
1The instructionBefore Answer only from retrieved passages; never state a rate, deadline, name or ID that isn't in front of you.
2Evidence verificationDuring Every "covered" claim must quote the source verbatim. The check collapses whitespace on both sides, so a hard-wrapped draft still matches. No verifiable quote → the claim is demoted and the quote dropped.
3The grounding verdictAfter Coverage is graded: at least 50% of the answer backed by retrieval passes. Otherwise regenerate once, strictly; if still weak, refuse.

What happens to a weak answer

Grade → exempt Lane 1 and personal recall → check the latency budget → regenerate once → refuse. If the turn has already spent its time budget (CHAT_LATENCY_BUDGET_S, 5 seconds in production), a usable first answer ships with a caution note instead of making the user wait for a second round-trip. Speed and caution are traded deliberately.

Measured, then deleted: an earlier fourth layer rewrote individual unsupported sentences. Log analysis showed it fired 0 times in 148 production turns, so roughly 220 lines were removed. The lesson is in the codebase: instrument the pipeline, then delete what never runs.

Caching: Layers, Lifetimes and Limits

Page 4 of 5

Caching exists to dodge two real costs — the Gemini call and a networked database read from a single-worker backend. The rule: cache the expensive, never the authoritative.

The chat answer cache — three tiers

All three sit behind one interface (query_cache.get/set), keyed on md5(normalized_query : model).

TierTechnologyLifetimeSizeScope
L1 — in-memorycachetools.TTLCache24 hours 500 entriesOne server instance
L2 — sharedRedis7 days— All instances; survives restarts. Degrades gracefully if Redis is down.
L3 — semanticGoogle text-embedding-004, 256-dim 7 days100 in memory Matches re-worded questions at cosine similarity ≥ 0.95.

L3 is the clever one: ask "what's the indirect cost rate" after someone asked "what's the F&A rate" and it recognises them as the same question. The threshold is deliberately high — 0.95 admits only near-identical meaning, preventing cross-topic false positives.

Citations ride along under a parallel cit: key, so a cache hit still shows its Sources.

What is never cached — and why it matters

  • Personal-recall answers. "What department am I in?" must never be replayed to somebody else.
  • Errors, outages and "busy, try again" messages. Caching one poisons the entry for its whole lifetime.
  • Refusals. Same reason.
  • Any personalised greeting. The cache key contains no user id, so an answer opening "Hello Maria!" would greet every later asker by that name. A leading salutation addressed to a proper name is stripped on both write and read.
The operational consequence: after editing a knowledge-base document, the old answer is still cached. Press Sync in the admin dashboard to clear the saved copies — otherwise you will re-ask, see the old answer, and conclude your edit failed. Edit → Save → Sync → re-ask.

Features and Scheduled Work

What the system actually does for a PI, and what it does on its own overnight.

The proposal toolkit

Each proposal carries a staged pathway — Set up → Build → Review & submit — with one recommended next action highlighted, so a first-timer is never staring at a flat toolbar of equal options.

ToolWhat it doesDeterministic core
Solicitation ingestion Upload the funder's PDF or paste a URL; extracts deadlines, page limits, budget caps, required attachments, eligibility — and seeds the checklist. Quotes verified against the source; fabricated quotes flagged.
Budget draft F&A, MTDC and fringe maths at Morgan's real rates, sponsor-cap check, multi-year projection with escalation, CSV export, drafted justification prose. All arithmetic in code. AI polishes wording only and cannot change a figure.
Drafting coach Per-section outline, then advisory feedback on the PI's own draft against that sponsor's review criteria. Coverage claims must quote the draft. Never writes prose for the author.
Compliance sentinel IRB, IACUC, conflict of interest, RCR training and export-control rules. Rule-based.
Draft critic Whole-package pre-submission check: page limits, required sections and attachments, budget against cap. A direct read of the PDF — no AI guessing.

Three standalone surfaces

Find Funding

Describe your research in plain English; returns live, open federal opportunities from the Grants.gov API across all 26 agencies, with a grounded fit explanation and a deterministic eligibility gate against Morgan's institutional profile.

Samples & Forms

19 hand-verified funded proposals (including six of Morgan's own awards), rankable against your research interests; plus the browsable ORA forms and templates catalog.

Memory — two tiers

Short-term is the current conversation. Long-term is a set of facts about the user — department, role, active grant, interests — consolidated overnight from chat. The saved profile is injected into every turn directly from the user record, so "what department am I in?" always works regardless of what memory selection returns. A hard PII gate screens the long-term write path.

Scheduled work — six internal endpoints

Cloud Scheduler calls these, authenticated by a shared secret header. Five recur; one is a one-off maintenance task.

The self-healing loop: a question the assistant cannot answer is logged as a genuine library gap (personal questions are excluded — a personal miss is not a gap). The overnight research job drafts a candidate document. An administrator then reviews, approves, and pushes it to the knowledge base. The machine may propose; only a person publishes.
