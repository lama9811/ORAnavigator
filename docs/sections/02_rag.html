2. How an Answer Is Made — the Chat & RAG Pipeline

This section explains, step by step, how one typed question turns into a trustworthy answer inside ORA Navigator. "Trustworthy" here has a specific meaning: the answer must come from ORA's own knowledge base (the collection of documents copied from the real ORA website), not from the AI's general memory of the internet. The whole pipeline exists to enforce that one promise.

First, two terms you will see constantly:

  • KB = "knowledge base." This is the set of 382 ORA documents (forms, policies, staff pages, compliance procedures) stored and searchable. The files live at backend/kb_structured/ and are indexed into a Google search index called a "datastore" named oranavigator-kb-v8.
  • RAG = "Retrieval-Augmented Generation." It means: before the AI writes anything, the system retrieves (looks up) the most relevant real documents, and the AI is told to generate (write) its answer using only those documents. "Augmented" = the AI's answer is backed up by real fetched text instead of its own memory.

The pipeline has three running pieces. The frontend (the web page, port 3001) sends your question. The backend (FastAPI, port 5002, file backend/vertex_agent.py) runs the safety checks. The ADK agent (port 8081, file adk_agent/ora_navigator_unified/agent.py) is the actual AI (Google's Gemini model) that does the searching and writing. The backend never fully trusts the AI — it checks the AI's work and can force a redo or a refusal.

2.1 The Quick Checks (no AI runs yet)

Before any AI is involved, three cheap checks try to answer instantly. Each one exists to avoid a slow, expensive AI call when the answer is obvious.

Check 1 — Greetings, thanks, and "who made this" (canned replies)

If you type something like "hi", "hello", "thanks", or "who made this app", a pattern-matcher catches it and returns a fixed, pre-written reply with zero AI calls. This lives in adk_agent/ora_navigator_unified/agent.py in the _greeting_fast_path function (agent.py:182), which is wired in as before_agent_callback.

Why: a greeting needs no facts, so paying for an AI call and a KB search would be wasteful and slow. It also avoids "context bleed" — for example after a chat about withdrawing a proposal, "who made this?" could otherwise confuse the AI into talking about forms.

Note: The "who made this app" answer is hard-coded so the bot never says "I don't have an app" — it always points to ora.inavigator.ai. A separate, load-bearing distinction: a plain greeting ("hi", "how are you", "thanks") gets a warm, KB-free reply — not the "I can only help with Morgan State…" refusal, and not the "developed for Morgan State / ora.inavigator.ai" identity blurb (that blurb is only for "who made this app"). The grounding gate must also never grade a greeting reply "weak" and regenerate it — that's what used to turn a friendly "I'm doing well!" back into a refusal (see Layer 3, Step A).

Check 2 — Was this exact (or very similar) question asked before? (the cache)

A cache is a memory of past answers. If the same question was already answered, replaying the stored answer is instant and free. ORA Navigator's cache has three tiers, all in backend/cache.py (class MultiTierCache):

  • L1 (in-memory): a small dictionary held inside the running server, about 0.001 ms to read. Holds up to 500 entries for 24 hours.
  • L2 (Redis): a separate shared database (Redis) so that all server copies share answers and they survive a restart, about 1–2 ms. In local development this is usually off, so only L1 runs.
  • Semantic cache: "semantic" means "by meaning." This tier turns each question into a list of numbers (an embedding) that represents its meaning, then matches questions that mean the same thing even if worded differently — e.g. "what is the F&A rate" matches "what indirect cost rate does Morgan charge." It only counts as a match above 0.95 cosine similarity so near-identical only — this avoids serving the wrong topic's answer.

The lookup order is L1 → L2 → Semantic → (miss = go ask the AI).

Note (load-bearing): Some questions must never be cached, because the answer depends on who is asking. The cache key is built from the question text plus the model name only — it has no user ID — so a personalized answer cached for one person would be wrongly served to everyone. The _should_cache gate blocks any question containing personal words like "my grant", "my deadline", "am i", "remind me", "about me", and it also defers to the same personal-recall detector the AI layer uses (_is_personal_recall) so the two layers agree on what "personal" means. Refusals and "I don't have that" answers are also never stored so a temporary refusal can't get stuck poisoning the cache.

Check 3 — "List X" / "What's in X" (instant menu from the KB tree)

If you ask to browse the knowledge base — "list the IACUC SOPs", "what's in pre-award", "show me your forms" — the system answers from a pre-built map of the KB instead of asking the AI. This is in backend/kb_browser.py, function try_browse. It reads a bundled file _manifest.json that mirrors the ORA website's left-side navigation (9 sections) and returns a formatted list in about 5–10 ms, with no AI and no search call.

To decide whether a question is really a "browse" request, it splits trigger phrases into two strengths:

  • STRONG triggers — unambiguous browse intent like "list", "enumerate", "browse", "show me", "what's in". Always eligible.
  • WEAK triggers — phrases that only look like browsing in isolation, like "give me", "tell me about", "what forms". On a follow-up turn (when there is already conversation history) a weak-only match is deferred to the AI instead, because mid-conversation "what forms do I need?" is usually a real question, not a request for a directory dump.

There is also a "topicless" safety rule: if the phrasing is browse-y but no specific topic matched, it only dumps the whole 9-section index for a STRONG request on a fresh turn — otherwise it defers to the AI. This fix stopped a real question like "what forms do I need to add a co-investigator?" from dumping the entire KB tree. A sibling function browse_citations returns the listed docs' titles and URLs so even these instant menus get a "Sources" block.

Why this lives in the backend, not the AI: Google's tool API will not let the AI use the KB-search tool and a custom "list the tree" tool at the same time. So the backend detects browse phrasing first and answers it deterministically — which also means these answers are impossible to hallucinate.

2.1b Two lanes — chatting versus asking

Not everything typed into a chat box is a question about ORA, and treating it as one used to produce something silly: you said "thanks!" and the assistant stapled a list of policy documents underneath its reply. So every incoming message is now sorted into one of two lanes before anything else happens.

Lane 1 — social chat. "Hi", "thanks", "how are you", "very good", and remarks aimed at the bot itself ("who are you?", "are you real?"). These get a warm, ordinary reply written by Gemini in its own words. Crucially, the knowledge base is not searched at all — the search tool is physically removed from the model for that turn (skip_kb_search), so it cannot attach sources to a greeting even if it wanted to. No Sources line appears, because none would be honest.

Lane 2 — a real ORA question. Searched, grounded, cited, fact-checked — the whole pipeline below. When a message could plausibly be either, it is treated as Lane 2 and searched. That default matters: erring toward looking it up is safe, while erring toward answering from the model's own memory is exactly how a wrong F&A rate reaches a PI.

Everything else is refused. "Write my cover letter", "who won the game", "explain photosynthesis" — a substantive question that isn't about ORA is politely declined. That is a deliberate boundary, not a gap in the assistant's knowledge, and staff should not report it as a bug.

The line is deliberately narrow. "Do you know my UEI?" and "what's my F&A rate?" sound personal, but they are institutional facts that live in the knowledge base — so they stay in Lane 2 and keep their Sources. Only genuinely personal identity questions ("what department am I in?", "who am I?") are answered from your profile instead, and those deliberately get no citations, because no KB document is where the answer came from.

2.2 Layer 1 — the TF-IDF keyword prefetch (a head start, still no AI judgment)

If none of the quick checks fired, this is a real ORA question and the AI will be called. But first, Layer 1 gives the AI a "head start" by stuffing some likely-relevant documents into the AI's instructions in advance.

TF-IDF (Term Frequency–Inverse Document Frequency) is a simple, no-AI scoring method that ranks documents by how often your search words appear in them, while down-weighting words that are common everywhere — so a rare, on-topic word like "IACUC" counts much more than a common word like "the." In one sentence: TF-IDF picks the documents whose wording best overlaps your question.

The code is adk_agent/ora_navigator_unified/kb_prefetch.py, function prefetch_kb_context. It keeps all 382 KB docs in memory (refreshed every 5 minutes), tokenizes your question into keywords (dropping filler "stopwords"), scores every doc, and returns the top 5. It also boosts ORA-specific terms: acronyms like IRB / IACUC / COI / F&A / FWA and SOP numbers get a big score bump so "SOP 12" finds the SOP page, not some unrelated text. This runs in under ~30 ms and costs nothing (no AI call).

That top-5 text is pasted into the AI's system instruction by the _select_model function in agent.py, which runs as before_model_callback. It is called "belt-and-suspenders" grounding: even if the AI forgets to run its real search tool, some real KB text is already sitting in front of it.

Note (the block is labeled "fact-check only" on purpose): the prefetched text is wrapped as [REFERENCE CONTEXT - FACT-CHECK ONLY] (it used to say "use this to ground your answer"). The relabel — paired with a new grounding rule (Rule 8, below) — tells the model the excerpts are a hallucination guard, not its answer: it must not answer from them alone and must not cite them. This was load-bearing. When the model answered straight from the prefetched block, its own VertexAiSearchTool retrieval never ran, so groundingMetadata came back empty — the answer was correct but had no Sources, and it tripped the weak-grounding verdict, firing a slow Pass-2 regeneration. With the relabel the visible answer and its Sources rest on the model's own KB search.
Note (gating — load-bearing bug fix): The prefetch is only injected on a fresh user turn, not in the middle of a tool call. The code checks in_tool_loop by looking only at the last item in the conversation: if it is a tool result (a function_response), the AI is mid-search and re-injecting prefetch would just push the real search results out of view. An earlier version scanned all past messages, so once any tool had been called, turn 1's leftover tool-result permanently disabled prefetch on every later turn — which left follow-ups like "give me a different training video" with no KB context and produced empty answers. Looking only at the last item fixes that.

2.3 The AI (Gemini) — search on every ORA question

Now the actual AI runs. It is Google's Gemini model (gemini-2.5-flash) driven through Google's "ADK" agent framework. Its full instruction sheet is BASE_INSTRUCTION in agent.py. The most important parts are the eight Grounding Rules.

The real retriever: Vertex AI Search (semantic = by meaning)

The AI is given one tool: Vertex AI Search (VertexAiSearchTool). This is the real retriever. Unlike the Layer-1 keyword prefetch, Vertex AI Search is semantic — it matches by meaning, not exact words. So if you ask "how long can I extend my grant past the end date", it can find the "no-cost extension" document even though your question never used the words "no-cost extension." Grounding Rule 1 orders the AI to call this search on every ORA-content question — rates, policies, staff, forms, deadlines, IDs — with no exceptions.

When the AI skips the search

The AI skips searching in two cases: (1) greetings and small talk (already caught earlier by the fast-path), and (2) "about me" recall. Grounding Rule 6 says facts the user stated about themselves (their department, role, active grant, deadlines) are not KB questions — those answers come from the chat history, not from a document search. Searching the KB for "what department am I in?" would find nothing and wrongly make the bot say "I don't have that information."

The eight Grounding Rules in plain English

  • Rule 1 — always search. Every ORA fact question triggers a KB search.
  • Rule 2 — never trust training data for Morgan facts. The AI's general internet memory is treated as outdated; only KB results count.
  • Rule 3 — never fabricate specifics. No making up names, emails, phone numbers, rates, dates, or IDs. If it's not in the KB results, "it does not exist as far as you know."
  • Rule 4 — honest deflection wording. When the KB has nothing or only partial info, the AI must say "Based on the information I have access to, …" and route the user to ORA at (443) 885-4044 / ask.ora@morgan.edu.
  • Rule 5 — enumeration uses the inventory tool first. For "list all X" type questions inside the AI path, it calls the deterministic topic list first.
  • Rule 6 — conversational recall. Facts about the user come from chat history, not the KB.
  • Rule 7 — NEVER return a blank answer. If asked to confirm a specific SOP / number / date that the KB doesn't contain, the AI must not go silent — it must explicitly say the item isn't in ORA's published records and route to ORA. A concrete example: "confirm IACUC SOP 37 covers cage sanitation." SOP 37 genuinely doesn't exist (the published list jumps 36 → 38). The AI must say so, never invent its contents. Before Rule 7, that question went silent and the user saw a useless error.
  • Rule 8 — "REFERENCE CONTEXT" is a fact-check only, not your answer. Any [REFERENCE CONTEXT - FACT-CHECK ONLY] block (the Layer-1 prefetch) is a hallucination guard. The AI must not answer from it alone and must not cite it — the visible answer and its Sources must rest on the AI's own live KB search. This rule, plus the relabeled block (see 2.2), is what restored real Sources to grounded answers.

The instruction sheet also repeats a hard "NEVER FABRICATE IDENTIFIERS" section that explicitly counts dates, page numbers, section numbers, and file names as identifiers too — so the AI can't paper over a gap by inventing "approved in 2017" or "see page 36" or "FringeRate-2018.pdf."

2.4 Layer 2 — the "receipt" (Sources)

When Gemini answers using its search tool, it returns grounding metadata alongside the text. Think of this as a receipt that says "these specific documents backed my answer." Each backing document is a grounding chunk (a chunk = one retrieved passage of KB text, with its title).

The backend reads this receipt in _do_agent_pass: it counts the chunks, and turns them into a clean, clickable "Sources" list via _extract_citations. Because the live search index stores the doc title but not its web address, the backend rebuilds a title→URL map from the local KB files (_get_kb_url_map) so each source resolves to a real morgan.edu link. It keeps only sources the answer actually cited, deduplicates them, and caps the list at 5.

The citation fallback (when a good answer comes back uncited)

Sometimes a trustworthy ORA answer arrives with no grounding receipt at all (Gemini's chunk metadata is unreliable, and it's often empty even for a perfectly grounded answer). To guarantee Sources, the DELIVER step runs _fallback_citations(query): it does its own live Vertex AI Search over the same unified datastore, resolves the matching titles → URLs via _get_kb_url_map(), and attaches them. It is gated by _wants_fallback_citations(), which skips small talk, refusals, outages, knowledge-base failures, and answers that are already cited — so it only adds real Sources to a trusted ORA answer that was missing them, and never to a greeting or a refusal.

Note (you can't just "force" the search tool): a natural question is "why not make the model always run its search tool?" Google's vertex_ai_search is native retrieval grounding, not a function the agent declares — and the tool_config mode=ANY knob that can force a tool governs only declared function_declarations. So forcing retrieval isn't possible; the relabeled prefetch (2.2), Grounding Rule 8, and this deterministic fallback are how Sources are guaranteed instead.
Note (Sources must survive a refresh): The Sources block is stored so it doesn't vanish when the page reloads or the answer is replayed from cache. There are two storage spots: (1) a citations column on the chat_history table — a JSON list of {title, url} — so reloading old chat history still shows Sources; and (2) a parallel cache entry under a cit: key so a cache hit can re-emit its Sources. This was the fix for Sources disappearing on cache/refresh/history reloads.

2.5 Layer 3 — the fact-checker (the real guarantee)

This is the part that makes the answer trustworthy. The backend does not simply trust the AI's draft. The orchestrator is _run_verified in vertex_agent.py. Here is what it does, in order.

Step A — run the AI (Pass 1) and grade the answer "ok" or "weak"

After Pass 1, the answer is graded by _evaluate_grounding. It is graded "ok" (deliver it) if any of these are true: it's a greeting/security/outage reply (no KB needed); it's already an honest "I don't have this" deflection; it was answered from an uploaded file or profile the user attached; it cited at least 2 KB documents; or at least 30% of the answer text is backed by KB passages. Otherwise it is graded "weak" and must be redone.

Two special cases bypass the grade. Personal-recall questions ("what's my deadline?" / "what department am I in?", detected by _is_personal_recall) flip a weak grade to "ok" — that answer correctly came from chat history, not the KB, and a KB-only redo would wrongly erase it. Greetings and small talk (detected by _is_smalltalk) do the same — a warm "I'm doing well!" must never be regraded weak and regenerated under the strict KB prompt, which would turn it back into a refusal.

Step B — honor the latency budget

An earlier step here was measured and then deleted — worth knowing why. The pipeline used to try a "surgical re-ground": rather than pay for a whole second answer, it found the individual sentences that were not backed by the retrieved text and rewrote only those. It was a good idea that never fired. A log analysis (scripts/analyze_layer3_logs.py) showed it triggered 0 times in 148 production turns, so roughly 220 lines were removed. In plain terms: answers were either well grounded or badly grounded, essentially never "90% fine with two stray sentences". Do not re-add it. What remains is simpler — if the answer is weak, the system either ships it with a caution note (when it has run out of time) or redoes the whole thing strictly.

There is also a hard latency budget. A turn that stacks a full Pass-2 regeneration on top of Pass-1 can blow past the <10-second target. So _over_latency_budget(elapsed) (controlled by CHAT_LATENCY_BUDGET_S, default 7 seconds) checks whether the turn has already spent its budget. If a usable (non-empty) Pass-1 answer graded weak and the budget is spent, the backend delivers Pass-1 with a short caution note instead of regenerating — skipping the second round-trip entirely. (An empty Pass-1 still regenerates; the streaming path already skips Pass-2 because the answer is already on screen.) A per-turn [LATENCY] log records the milliseconds, the verdict, and the chunk count so the <10s target is measurable.

Note (the bigger lever was warm instances): the latency budget trims the worst-case turn, but the dominant cause of a slow first answer was never the pipeline — it was the 30–60-second Cloud Run cold start. Keeping the ADK and backend warm (Chapter 1) is what actually removed the ~40-second waits; the budget guard is the secondary lever.

Step C — if still weak, redo once under a strict KB-only prompt (Pass 2)

If the answer is weak, the backend wipes the AI's session (so the weak draft isn't in the AI's view) and asks again, this time prepending a strict instruction called _STRICT_PREFIX. In plain words, that prefix tells the AI: answer only from the knowledge base; you may still use facts the user told you about themselves; never guess or approximate; and if the KB genuinely doesn't have it, say so and point the user to ORA. This is the "double-checking sources" step the user sees in the progress indicator.

Step D — trust the clean redo (don't re-gate on the flaky chunk count)

This is a subtle but load-bearing decision. The strict redo already forced the AI to either ground its answer or honestly deflect. So if the redo produces any non-empty text, the backend trusts it and delivers it — it deliberately does NOT re-check the grounding-chunk count again. Why: Gemini reports that chunk-count metadata unreliably (it's often empty even for a perfectly correct, grounded answer), and re-gating on it is exactly what used to make Layer 3 throw away good answers.

Step E — if the redo also produces nothing, REFUSE

If even the strict redo comes back empty or errored, the backend refuses rather than guess. It returns _REFUSAL_MSG: "I don't have reliable information on this in my knowledge base, so I'd rather not guess. Please contact ORA at 443-885-4044 or ask.ora@morgan.edu …" A related edge case: if Pass 1 returns empty text and zero chunks (the dead-end that used to fire a useless "try rephrasing" error), it now degrades to this same honest refusal.

Step F — the identifier faithfulness check (verify the specifics word-for-word)

For an answer that passed, _finalize_answer runs one more guardrail: _check_identifier_faithfulness. It scans the answer for high-risk specifics — SOP numbers, FWA / EIN / UEI numbers, F&A and fringe percentages, dates, dollar amounts, emails, phone numbers, page / section numbers, and file names — and checks each one appears verbatim (exact, word-for-word) in the retrieved KB text. If a number was invented (not found in the docs), it's logged. Two safeguards prevent false alarms: ORA's own main phone and email are whitelisted so canned messages don't flag themselves, and it is negation-aware — it skips a value the answer is refuting ("the rate is NOT 99%"), since refuting a wrong number is faithful behavior, not a fabrication.

Note (the removed "could not be verified" footer): There used to be a user-facing footer that appended "I couldn't verify this number against ORA's records…" whenever a specific value wasn't found verbatim. It was removed on 2026-06-02 because it produced false positives: the check is purely token-based and couldn't tell asserting a number from refuting one — so when the bot correctly rejected a planted fake "99%" F&A rate, the footer still fired and flagged the very number the bot was denying. The separate staff-name faithfulness disclaimer stays active — it appends a "verify at the staff directory" note when the answer mentions a "Dr. X" whose surname isn't in the ORA roster.

2.6 The load-bearing bug fixes that keep this reliable

Each one quietly keeps the pipeline correct; removing any of them re-introduces a real, already-seen bug.

FixWhat it does, in plain English
/run_sse Bearer token When the backend calls the AI service on Google Cloud, that service is private and requires a login token. _get_auth_headers() attaches a "Bearer" ID token to the /run_sse call. Without it, Cloud Run rejects the request with a 403 "forbidden" error and no answer is produced.
GOOGLE_GENAI_USE_VERTEXAI=TRUE This environment variable tells Google's AI library to use the company's Vertex AI service (which the project is set up for). Without it, the library falls back to the public consumer Gemini API and demands a separate GOOGLE_API_KEY that the project doesn't use — so the AI call fails.
Single-worker server + thread-local grounding state The grounding "receipt" for the current request (chunks, citations) is kept in a per-thread slot. This is safe only because the server runs as a single worker process. If someone scaled the server to multiple workers without rethinking this, one user's Sources could leak onto another user's answer.
Cache keys include the user, and refusals/personal questions aren't cached Personal-recall answers and refusals are never written to the cache, using the same _is_personal_recall detector the AI uses. This stops one person's private answer (or a temporary refusal) from being replayed to everyone else who asks a similar question.
Prefetch gating The Layer-1 prefetch only fires when the last message is a real user turn, not mid-tool-loop. Scanning all past messages instead of just the last one once permanently disabled prefetch after the first tool call — leaving follow-up questions with no KB context.
Model name is gemini-2.5-flash everywhere The model IDs must be gemini-2.5-flash. The older gemini-2.0-flash name returns a 404 "not found" in this Vertex project, so using it breaks every answer.
Frontend SSE buffer flush The answer streams to the browser line-by-line. The browser's reader splits on newlines and keeps the last unfinished line in a buffer; it must flush that buffer after the stream ends. The final data: {done} frame can arrive without a trailing newline as the connection closes; if the buffer isn't flushed, that frame is dropped and the message stays "still streaming," firing the misleading frontend-only fallback "I couldn't generate a response." That string is a frontend fallback — the backend always produced and saved a real answer. Don't remove the flush.
Gemini 429 arrives as text, not an exception When Vertex/Gemini is over quota, the error comes back as response text ("429 … RESOURCE_EXHAUSTED"), so it slips past the normal except handlers. _is_rate_limited detects it and retries with backoff, then flags a real outage — instead of laundering it into a "system is busy" sentence and then amplifying the load by regenerating it in Pass-2.
Grounded citations from the real KB search A correct-but-uncited answer was the symptom of the model answering from the prefetched block instead of its own retrieval. The fix has three parts: the prefetch is relabeled [REFERENCE CONTEXT - FACT-CHECK ONLY] (2.2); Grounding Rule 8 forbids answering from or citing it; and _fallback_citations runs a live KB search to attach real Sources when a trusted answer came back uncited (2.4). Forcing the native retrieval tool isn't possible, so these three together guarantee Sources.
Latency budget + warm instances A turn can stack a full Pass-2 regeneration on Pass-1 and blow the <10s target. _over_latency_budget (CHAT_LATENCY_BUDGET_S, default 7s) delivers a usable Pass-1 + caution note instead of regenerating once the budget is spent, and a [LATENCY] log makes the target measurable. The bigger lever, though, was keeping the ADK and backend warm so the chat path never pays a 30–60s cold start.
Token-by-token streaming with a guard The answer streams cumulatively from the agent; the backend emits the new delta each event so the user sees it appear word-by-word. A short prefix guard (the first 60 characters) is held back before anything is shown, so a leaked error (a 429 or a KB-access failure) is caught and never streamed to the user. The streaming path never regenerates Pass-2 — the answer is already on screen — it only appends the caution note when weak.
Big-picture takeaway: The system answers cheaply when it can (canned replies, cache, instant KB menus), gives the AI a real head start (Layer 1 prefetch), forces the AI to search real documents on every fact question (Gemini + Vertex AI Search), shows its receipts (Layer 2 Sources, saved to chat_history.citations), and — most importantly — never blindly trusts the AI: Layer 3 grades the answer, redoes it strictly once if it's weak, refuses rather than guess if it still can't ground it, and flags specific numbers that don't appear word-for-word in the source documents.
