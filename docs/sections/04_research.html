4. The Self-Healing Research Pipeline & Ticket System

This section explains two connected systems inside ORA Navigator. The first is the Self-Healing Research Pipeline (the code calls it the "Auto-Research Agent"). The second is the Ticket / Feedback / Support system. They are connected because when a user "reports" a bad answer, that report is fed back into the research pipeline.

4.1 The Problem It Solves

First, two plain-English definitions:

  • Knowledge Base (KB): the collection of documents the chatbot is allowed to read when it answers. ORA Navigator's KB has 382 documents scraped from the official Morgan State research-administration website. The bot is only supposed to answer from these documents (this is called "grounding") so it does not make things up.
  • KB miss: a question the bot could not answer because the KB simply does not contain that information. Example: a user asks "What time does the IRB committee meet in July?" but no document mentions that meeting time.

Before this pipeline existed, a KB miss just vanished. The user got "I don't have information about that," and nobody ever found out the bot had a gap. The knowledge base never got better.

The Self-Healing Research Pipeline fixes that. The word self-healing means the system notices its own gaps and repairs them. Every time the bot fails to answer, the question is recorded. Similar failures are grouped together. A research robot goes and finds the real answer on the official Morgan State website. Then a human admin reviews that answer and, if it is correct, publishes it into the live knowledge base. The next person who asks the same question gets a real answer. The "wound" (the gap) heals.

Note: There is always a human gate. The robot never publishes anything to the live KB on its own. An admin must click "Approve" and then "Push." This prevents a wrong AI-researched fact from ever reaching real users automatically.

4.2 The Five Stages

The pipeline moves a failed question through five stages. Each stage changes a status value in the database and saves the work in a specific place. The two database tables involved are failed_queries (the FailedQuery class) and kb_suggestions (the KBSuggestion class).

Stage 1 — Detect (saved to failed_queries, status "new")

The detector is the function detect_and_log_failed_query. It runs automatically after the bot answers a chat message. It is called from three places: the /chat endpoint, the streaming chat endpoint, and the guest chat endpoint.

The detector decides "was this a KB miss?" using two detection layers:

  • Layer 1 — Grounding count (the main signal): The bot reports how many KB document chunks it actually used to write its answer. This count comes from get_last_grounding(). If that number is zero — meaning the bot found nothing in the KB to stand on — it is treated as a miss. A "chunk" is just a small slice of a KB document. Zero chunks = the bot was answering from nothing.
  • Layer 2 — Failure language (the backup signal): Sometimes the KB returns some documents but none are actually relevant, so the bot writes a sentence like "I don't have information about that." Layer 2 checks whether the first line of the answer starts with that kind of giving-up language. It only matches when the failure language is the first thing said, so a good answer that merely hedges one small detail is not wrongly flagged.

Before either layer runs, the detector skips several kinds of messages so the failed-query list does not fill up with junk:

  • Greetings and tiny messages — "hi", "hello", "thanks", "ok", etc., and anything shorter than 15 characters.
  • Security refusals — if the answer starts with "I can only help with Morgan State…" the user asked something off-topic or tried a prompt injection. That is not a knowledge gap.
  • File uploads — if the message contains a markdown file link, it was a PDF the user uploaded for analysis, not a KB question.
  • Short conversational follow-ups — a message of 6 words or fewer that has no real question word ("what", "where", "who", "when", "how", "why", "which", etc.).

If it survives all the skips and is judged a miss, the detector writes one row into the failed_queries table with the user's question in user_query, the bot's reply in bot_response (capped at 1000 characters), the user's id, and status = "new". It first checks for an identical existing question so the same query is not logged twice.

Stage 2 — Cluster (saved to failed_queries, status "clustered")

Many people ask the same thing in different words: "What is the F&A rate?", "What's Morgan's indirect cost rate?", "How much overhead does Morgan charge?" — three questions, one answer needed. Clustering means grouping these near-identical questions together so the robot only has to research the topic once. The function is cluster_failed_queries.

How it groups them, in plain words:

  • It takes all the "new" failed queries that have not been clustered yet.
  • It turns each question into an embedding — a list of 256 numbers that represents the meaning of the sentence. It uses the text-embedding-004 model. Two sentences that mean the same thing produce two very similar number-lists.
  • It compares every pair of questions using cosine similarity — a score from 0 (totally different meaning) to 1 (identical meaning). If two questions score 0.82 or higher, they are put in the same group.

Each group gets a shared cluster_id number, and every query in the group is updated to status = "clustered". Both values are saved back onto the same rows in the failed_queries table.

Stage 3 — Research (saved to kb_suggestions, status "pending")

Now the robot actually goes and finds the answer. The function is research_topic. It uses Gemini (Google's AI model, gemini-2.5-flash) with the Google Search tool turned on. "Grounding" here means the AI is told to base its answer on real web search results, not its own memory.

The prompt tells Gemini to search the official Morgan State website (morgan.edu) and especially the Office of Research Administration pages, to find specific facts (names, dates, phone numbers, URLs, deadlines, SOP numbers, form IDs), and to return a strict JSON object containing: a short topic label, the answer, a list of sources (URLs), a confidence level ("high", "medium", or "low"), a suggested_doc_id (which KB document to add to), and the suggested_content (the text ready to drop into the KB).

The batch orchestrator run_research_batch loops over the clusters, calls research_topic for each one, and saves the result as a new row in the kb_suggestions table with status = "pending". It also flips every failed query in that cluster from "clustered" to status = "researched" so it is not researched again. It skips any cluster that already has a suggestion.

Stage 4 — Review (the human gate; status becomes "approved" or "rejected")

A human admin opens the Research tab in the Admin Dashboard and sees every pending suggestion as a card showing the topic, confidence, how many users asked, the researched answer, its sources, and an editable text box with the content that will be pushed. The admin can fix the text before approving.

The admin clicks Approve or Reject. The browser sends a PUT request to /api/admin/research/suggestions/{id} handled by review_suggestion:

  • Approve → status = "approved", and it records reviewed_by and reviewed_at.
  • Reject → status = "rejected", saving the admin's reason in admin_notes.
  • Edit → saves the admin's changed suggested_content / suggested_doc_id / notes without changing the status. The "Approve" button actually sends an edit first (to capture the admin's textbox changes) and then the approve.
Note: Nothing has touched the live knowledge base yet. An "approved" suggestion is just a green light. The actual publishing happens in Stage 5, behind a second button and a confirmation dialog.

Stage 5 — Push (written into the live Vertex AI Search datastore; status "pushed")

Vertex AI Search datastore is Google's hosted search index that holds the live 382-document knowledge base the chatbot actually searches. Pushing means writing the new content into that live index so the chatbot can immediately find it.

On an approved card the admin sees a "Push to KB" button. Clicking it shows a confirmation pop-up ("Push this suggestion to the live knowledge base?") and then sends a POST to /api/admin/research/suggestions/{id}/push, handled by push_suggestion. That function:

  • Loads the suggestion, but only if it is already approved — you cannot push something that was never reviewed.
  • Checks whether the target document already exists. If it exists, it appends the new content to the end of the existing document; if not, it creates a brand-new document.
  • On success, sets status = "pushed" and clears the caches — the query cache and the agent's session cache — so the chatbot does not keep serving a stale "I don't know" from memory. The next identical question is answered fresh from the freshly updated KB.

After this, kb_suggestions.status = "pushed" and the gap is healed.

4.3 What the "Run Research Now" Button Actually Does

In the Research tab there is a "Run Research Now" button. Its handler is handleRunResearch, which sends a POST to /api/admin/research/run. That endpoint is trigger_research (admin-only), and it calls run_research_batch().

In plain words: this button runs Stage 2 (Cluster) and Stage 3 (Research) only. It groups all the new failed queries and has Gemini research each group, which fills the review queue with new pending suggestions. When it finishes, it shows a toast like "Clustered 8 queries, researched 3 topics".

Note: "Run Research Now" pushes nothing to the live KB. It stops at "pending." A human still has to approve and push each suggestion. The button only prepares work for the admin to review.

The automatic trigger: the same research batch also runs on a schedule with no human present. Cloud Scheduler (Google's cron service) sends a POST to /api/internal/research/run, handled by internal_research_trigger. Because no admin login is attached, this endpoint is protected by a shared secret: the caller must send an X-Research-Secret header that matches the server's RESEARCH_SECRET environment variable, otherwise it is rejected with a 403 error. It then runs the exact same run_research_batch() — so the queue keeps filling even overnight. Approval and pushing always remain manual.

4.5 The Ticket / Feedback / Support System

This is the second system. It is how users tell the team about problems, and it is also the bridge that feeds bad answers back into the research pipeline above.

Support Tickets

A support ticket is a formal help request — a bug report, a feature idea, or a question — that a user files and an admin works through to resolution. Tickets are stored in the support_tickets table (the SupportTicket class). Its columns:

  • user_id — who filed it.
  • subject — the short title.
  • category — "bug", "feature", "question", or "other".
  • description — the full text of the problem.
  • attachment_data / attachment_name — an optional file (e.g. a screenshot), stored as Base64 text up to ~12 MB.
  • status — "open", "in_progress", "resolved", or "closed" (default "open").
  • priority — "low", "normal", "high", or "urgent" (default "normal").
  • admin_notes, resolved_by, resolved_at — the admin's reply and who closed it.
  • created_at / updated_at — timestamps.

How a user files one: the app sends a POST to /api/tickets, handled by create_ticket. It trims the subject/description, caps the attachment at ~7.5 MB, inserts a new SupportTicket row, and returns the new ticket id. Where it is stored: the support_tickets table in Cloud SQL.

How admins see and handle tickets: the Admin Dashboard has a Tickets tab that loads them from GET /api/tickets (admins see all tickets; a normal user sees only their own). Admins change a ticket's status or add notes with PUT /api/tickets/{ticket_id} → update_ticket, which is admin-only and, when the status is set to "resolved," records resolved_by and resolved_at.

Feedback (thumbs up / thumbs down / report)

Feedback is the lightweight one-click reaction on a single chatbot answer. It is stored in the feedback table (the Feedback class). Its columns:

  • user_id, session_id — who reacted and in which chat session.
  • message_text — the text of the bot answer being rated.
  • feedback_type — one of "helpful" (thumbs up), "not_helpful" (thumbs down), or "report" (an explicit "this answer is wrong / missing info" report).
  • report_details — optional extra text the user types when reporting.
  • timestamp.

How a user leaves feedback: the app sends a POST to /api/feedback, handled by submit_feedback. It validates that the type is one of the three allowed values and inserts a Feedback row into the feedback table. How admins see it: the Admin Dashboard's Feedback tab reads /api/feedback/stats and /api/feedback/all, which return totals (helpful / not_helpful / reports, a satisfaction rate) and the recent reports. A red badge on the tab shows the number of open reports.

The bridge: "Report a bad answer" feeds the research pipeline

This is the important connection between the two systems. When a user clicks Report on an answer (feedback_type = "report"), it means "this answer is factually wrong or missing information" — which is exactly a knowledge-base gap. So submit_feedback does an extra step:

  • It looks up the original chat turn in chat_history by matching the reported answer text.
  • If found and not already recorded, it inserts a new row directly into the failed_queries table with status = "new" — the very same table Stage 1 of the research pipeline writes to.

So a user-reported bad answer enters the self-healing loop at Stage 1 and will be clustered, researched, reviewed, and pushed just like an automatically detected miss.

Note: A plain thumbs-down ("not_helpful") is not turned into a failed query. The reason: people thumb-down for many reasons — too long, wrong tone, didn't like the format — that do not mean the knowledge base is missing a fact. Only an explicit "report" is treated as a real KB gap and pushed into the research pipeline.
