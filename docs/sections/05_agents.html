5. The Proposal Toolkit — AI Agents, Tools & Forms

This section explains the part of ORA Navigator that helps a faculty member actually build and submit a grant proposal — not just chat about grants. It is a whole toolkit: some pieces are real LLM-backed "agents," others are deterministic tools with an optional advisory AI layer, plus the screens and database tables they read and write. Everything here is wired up so a wrong guess by the AI can never quietly cause harm: a human always reviews before anything is committed, the deterministic core is authoritative, and any AI claim is double-checked against the real document.

5.1 The big idea: tools that coordinate through a shared record

A "tool" here just means a focused piece of software that does one job; an "agent" is a tool that can call the AI model (Gemini) when it needs to. The proposal toolkit is:

  • Solicitation Ingestion — reads a sponsor's grant announcement (a PDF or a pasted URL) and pulls out the key facts.
  • Budget Helper — a deterministic grant-budget calculator (F&A / MTDC / fringe math) with an AI-polished justification narrative.
  • Drafting Coach — per-section outlines and advisory feedback on the PI's own draft prose.
  • Compliance Sentinel — a deterministic "which approvals do I need?" questionnaire (IRB / IACUC / COI / RCR / export).
  • Draft Critic — checks the whole draft package against the rules before submission.
  • Deadline Watcher — emails faculty as a deadline gets close.
  • Task Guidance — a plain-English how-to attached to each checklist task.

There are also two standalone discovery surfaces (browse pages, not per-proposal tools) covered later in this chapter: the Opportunity Finder (find live federal grants) and the Sample Proposals Library (read worked examples).

Note (what was removed): Two tools were deleted by product decision and must not come back. The "Sponsor Fit-Finder" agent was removed on 2026-06-01 (the funding knowledge-base documents stayed, so the chatbot still answers funding questions). The "Fundability / Eligibility self-check" tool (reviewer-lens scoring + go/no-go) was also removed — its files (services/fundability.py, services/eligibility.py), tests, the FundabilityModal, and the /fundability · /eligibility endpoints are all gone. The eligibility text extracted from a solicitation is still kept (_eligibility_text_from_notes) and fed to the Drafting Coach.

The most important design point: the tools never call each other directly. Instead, they all read and write the same proposal record in the database. Think of it like a shared notebook on a desk. Solicitation Ingestion writes the proposal's facts into the notebook. Later, Budget Helper, the Drafting Coach, the Compliance Sentinel, and Draft Critic each open the same notebook to learn the rules. Deadline Watcher opens it to see when things are due. None of them phones the others — they just share the notebook (the submissions and submission_tasks tables in the database).

The AI-backed pieces are protected by evidence verification, not just polite instructions. "Evidence verification" means: after the AI says something, the code mechanically checks whether the AI's claim is actually backed by a real quote from the source document. If the AI invents a fact, the code catches it and throws it away. This is the difference between asking the AI to behave ("please don't make things up") and forcing it to ("if you can't quote it, we delete it"). The deterministic tools (Budget Helper math, the Compliance Sentinel) have no AI in the authoritative path at all.

The shared Gemini client

Every AI-backed piece reaches Gemini through one shared helper, backend/services/gemini_client.py — "one reusable phone line to Gemini 2.5 Flash." It does three helpful things:

  • Builds the connection once and reuses it — no slow setup on every call.
  • Fails fast and safely. If the AI can't be reached at all (for example, running offline or in a test), every call returns None immediately instead of hanging. This is what lets each tool quietly fall back to its non-AI behavior.
  • Never crashes. The two public functions generate_text and generate_json always return either a clean result or None — they never raise an error. generate_json also strips out any stray markdown code-fences and tolerates odd control characters that PDFs sometimes leak.

The model name is hard-coded to "gemini-2.5-flash" because the older gemini-2.0-flash name returns a "not found" error in this Google Cloud project.

5.2 Solicitation Ingestion (a PDF or a URL in, a structured proposal out)

A "solicitation" (also called an FOA, "Funding Opportunity Announcement") is the long document a sponsor like NSF or NIH publishes to describe a grant: who can apply, how much money, the deadline, page limits, what attachments are required. Reading it carefully is tedious and error-prone. This agent does the first pass for you. Code lives in backend/services/solicitation_extractor.py and backend/services/url_fetcher.py.

Two ways in: upload a PDF, or paste a URL

There are two entry points that both end at the same extractor:

  • Upload a PDF — POST /api/me/submissions/from-solicitation. The file must be a PDF and at most 25 MB. The PDF is read into memory only (pdf_bytes = await file.read()), parsed locally with pdfplumber, and discarded — it is never saved to disk or any table. Because it is 100% local, an uploaded PDF always works, even for funder sites that block servers.
  • Paste a URL — POST /api/me/submissions/from-solicitation/url (body { "url": "..." }). The backend fetches the page itself with fetch_solicitation_text and feeds the text to the same extractor, so the review screen is identical. The Opportunity Finder's "Start a proposal from this" handoff lands here (it pre-fills the URL box via the modal's initialUrl prop).

Fetching a URL safely (SSRF guard)

Letting a server fetch a user-supplied URL is dangerous — a malicious URL could point at the cloud's internal metadata service or a private IP. So url_fetcher.py is locked down:

  • It resolves the hostname (_resolve_public_ip) and rejects any private, loopback, link-local, reserved, multicast, or unspecified address (_ip_is_public). If any resolved address is internal, the whole request is refused — including the cloud metadata endpoint 169.254.169.254.
  • It then pins the validated IP (_PinnedIPAdapter) so the actual connection goes to the address that was just checked — defeating "DNS rebinding," a trick where a hostname resolves to a safe IP during the check and a private IP a moment later. The real hostname is still used for the TLS certificate and Host header. Each redirect hop (up to _MAX_REDIRECTS = 5) is re-resolved and re-validated.

When funder sites block the server (the scraper fallback chain)

Some funder sites (NSF and other Akamai-fronted sites) deliberately block requests coming from cloud-datacenter IP addresses, so a direct fetch from Cloud Run gets a 403/blocked. To handle that, fetch_solicitation_text tries the direct fetch first, and on a blocked/empty result falls through a chain of scrapers in order:

  1. Firecrawl (_fetch_via_firecrawl, preferred) — a hosted service that runs a real browser from non-blocked IPs. It needs FIRECRAWL_API_KEY, and it treats the literal placeholder value "unset" (the deploy self-heals a missing secret to this) as "no key."
  2. Jina reader (r.jina.ai) — a public read-through proxy; an optional JINA_API_KEY lifts its rate limit.
  3. allorigins — a no-key public proxy, last resort.

The fetched page is converted to text locally (HTML via lxml, or a downloaded PDF via pdfplumber), capped at 25 MB to match the upload limit.

Gemini extracts the facts as JSON

However the text arrived, it (up to 250,000 characters — a big window so facts buried deep in long FOAs aren't missed) is sent to Gemini with a very strict instruction sheet called _EXTRACT_SYSTEM. Gemini must return a JSON object with these fields: sponsor, program_id, program_name, deadline, deadline_details, page_limits, required_attachments, eligibility, budget_cap, budget_cap_details, submission_portal, and source_quotes.

The source_quotes field is the clever part. For every value Gemini fills in, it must also paste the exact verbatim sentence (≤200 chars) from the document that proves that value. If Gemini can't find a real supporting quote, the rules tell it to return null ("no quote, no value"). The instruction sheet also forces other safety rules: never guess a deadline, use the most restrictive number when several are given (smallest budget cap, smallest page limit, so an applicant is never told they have more room than they do), and budget_cap means the maximum per award / per proposal, not the total program budget.

Multiple deadlines and per-category caps

Real solicitations often have several deadlines or several funding tracks with different award maxima (e.g. NSF IDSS Category I / II / III). Two extra fields capture that without losing the single safe number:

  • deadline_details — a plain-text summary listing each deadline with its category/condition, surfaced in an "All deadlines (by category)" box. The single deadline stays the one used for reminders.
  • budget_cap_details — an array of {category, cap} when tracks have different maxima (a stated range is read as its maximum). The single budget_cap stays the smallest cap, so the default is conservative. These per-category caps are persisted as a Category caps: line in the proposal's notes and round-tripped by reconstruct_solicitation_context; the Budget Helper then shows a "Funding category" picker (see 5.3).

The anti-hallucination check: flag, don't drop

Telling the AI "always quote your source" is not enough — an AI can paste a fake quote. So after Gemini answers, a deterministic (non-AI) function _verify_source_quotes checks each filled field: it normalizes the text (lowercase, drop bullet symbols, collapse spaces) and asks "is this quote actually a substring of the real document text?" If a field has a value but no real quote behind it, that field name is added to a list called unverified_fields.

The key policy is "flag, don't drop": the suspicious value is not deleted. It is kept but marked, and the UI shows it in red (with a ⚠ "couldn't back this with a quote" note) so the human reviewing it knows to double-check that one. This is safe because a human reviews everything before it becomes a real proposal.

Note: A wrong deadline can cost a faculty member an entire grant. That is exactly why this agent never creates a proposal by itself. Step one only returns the extracted facts for review. Nothing is saved yet.

Step two: the human confirms, and a Submission is born

After the user reviews and edits the extracted fields on screen, the browser calls a second, separate endpoint, POST /api/me/submissions/from-solicitation/confirm. This two-step design means a wrong AI guess can never auto-commit — the user must explicitly confirm.

Confirm calls create_submission_from_solicitation, which is where the extracted facts finally get saved to the database:

  • A new row is inserted into the submissions table: the title, sponsor, parsed deadline, status="active", and a notes text blob.
  • The notes column carries the leftover facts in a structured, machine-readable format — lines like "Budget cap: $600,000", "Category caps: Category I — $750,000; ...", "Page limits: project_description: 15p, ...", "Required attachments: ...", and "Eligibility: ...". This matters later: the Budget Helper, Drafting Coach, and Draft Critic all read these exact lines back out.
  • The checklist tasks are inserted into the submission_tasks table: first the sponsor's standard template tasks, then one extra task per required attachment that the template didn't already cover (titled "Prepare required attachment: X").
ArtifactWhere it is saved
The uploaded solicitation PDF (or fetched URL text)Nowhere persistent — read into memory, used, then discarded. Never on disk, never in a table.
The extracted fields (before confirm)Nowhere — returned in the HTTP response only; they live in the browser until the user confirms.

5.3 Budget Helper (deterministic grant-budget math, AI-polished prose)

The Budget Helper is the trustworthy, deterministic core of the toolkit. A PI types in line items — people (base salary, % effort, fringe category), plus equipment, travel, supplies, participant support, "other," and subawards — and the modal computes a full federal grant budget live as they type. Code lives in backend/services/budget_helper.py; the UI is frontend/src/components/BudgetHelperModal.jsx. Every number is computed by code; the LLM is forbidden to change any figure.

The deterministic math

The entry point is compute_budget. It applies Morgan State's real, hard-coded rates:

  • F&A (indirect cost) rates — for FY 2025–2026: Organized Research (on-campus) 54%, Instruction 64%, Other Sponsored Activity 42%, Off-campus 26% (the prior year's organized-research rate was 53%). The default is organized research on-campus.
  • Fringe rates — Faculty (academic year) 42%, Faculty summer 9%, Full-time / non-contractual 42%, Contractual 9%.

The key subtlety it gets right: F&A is charged only on the Modified Total Direct Cost (MTDC) base, not on every dollar. MTDC = total direct costs minus equipment, minus participant support, minus the portion of each subaward above the first $25,000. Per person, salary = base × effort% and fringe = salary × fringe_rate; the F&A amount is MTDC × F&A rate; the grand total is direct + F&A. It then checks the total against the sponsor cap and reports cap_status as none / ok / over. Junk and negative inputs are coerced to zero with a warning; effort over 100% is clamped.

The coaching layer (advisory; never changes a number, never blocks saving)

  • budget_advisories() — sanity flags: equipment over 40% of direct costs, travel over 25%, a person with salary but 0% effort, effort but $0 base, no personnel at all, or subawards over half the budget.
  • suggest_trims() — only fires when you're over the cap. It proposes concrete cuts to the flexible F&A-eligible lines (travel → supplies → other); because those sit in MTDC, every $1 cut saves $(1 + F&A rate) of total cost, which the suggestion explains.
  • Plain-English tooltips for each category, the F&A choice, and fringe.

v2: multi-year, CSV, and a real justification

  • Multi-year projection. If the inputs include project_years > 1, compute_budget escalates each person's salary by the chosen escalation percent each year, computes every year, and reports per-year and cumulative totals — and checks a multi-year award's cap against the cumulative total. Single-year output is unchanged.
  • CSV export. budget_to_csv serves a download at GET /api/me/submissions/{id}/budget.csv; its row order mirrors the on-screen grid.
  • Deterministic prose justification. draft_justification writes a labeled-paragraph budget-justification narrative (Personnel / Other direct costs / F&A / Total project cost) using full sentences, every figure pulled straight from the computed budget — it reads like a real submission. There is also a one-sentence per-line justification.
Note (AI is forbidden to touch figures): Only one endpoint, POST /api/budget/justification, calls the LLM, and only to rewrite the prose — the prompt explicitly says "do NOT change any dollar figure, percentage, name, or rate." There is a truncation guard: the AI output is kept only if it is non-empty and contains the total-cost figure verbatim; otherwise the code logs the truncation and falls back to the deterministic template. So a complete, figure-correct justification always comes back even when Gemini is down or cut off — a known under-load failure mode where the box would otherwise show a half-sentence.

Read-only effort helper, the category picker, and the spreadsheet view

  • The effort → person-months helper is read-only: it derives the months from each person's effort_pct in the People rows (academic 9 mo, summer 3 mo, calendar 12 mo), so it can never disagree with the budget.
  • When a solicitation defined two or more funding categories, a "Funding category" picker appears (and only then). Its default is a blank "Select your category…" so the PI consciously chooses; picking a category fills the sponsor cap from budget_cap_details.
  • compute_budget also returns a render-ready table ({columns, rows}; one "Amount" column for single-year, a column per year plus "Total" for multi-year), built by _budget_table from the per-year breakdowns. The modal renders it as a read-only Budget table grid that formats numbers but computes none, so it always matches the summary and the CSV.
Note (a load-bearing CSS detail): the Budget table's <table> uses the class bh-table — not bh-grid, which is the form's two-column display:grid and would silently turn the table into a broken 2-column grid with the header and body split and amounts clipped. The whole modal body (form + summary + table + justification) scrolls as one .bh-scroll region; the navy Summary is position:sticky so the total stays in view while the taller form scrolls (disabled on mobile). The justification panel has a Close (×) that clears the draft without closing the whole modal.

5.4 Drafting Coach (section outlines and advisory feedback)

The Drafting Coach helps a PI write one proposal section at a time. It has two modes and never writes the prose for the PI — it is coaching only. Code lives in backend/services/section_coach.py; the UI is SectionCoachModal.jsx.

  • Outline mode — a deterministic skeleton of the must-have elements for that section (always returned), optionally tailored by AI to this funder.
  • Feedback mode — advisory review of the PI's own pasted draft: which required elements are covered, partial, or missing, each backed by a verbatim quote from the draft.

The section menu is by sponsor

The constant _SPONSOR_ORDER picks which sections to show: NSF → Project Summary / Project Description / Broader Impacts / Data Management Plan; NIH → Specific Aims / Research Strategy / DMP; any other sponsor → a generic Abstract / Narrative / DMP.

What it does (v2 and v3)

  • Save drafts — a PI's section text is stored in the submissions.sections_json column (endpoints GET/PUT .../sections/drafts).
  • Live length meter — the WORD_TARGETS constant gives each section a sensible word range (e.g. Project Summary 200–500), and feedback flags a draft as "long" or "short."
  • Deterministic clarity check — clarity_check() (no AI) flags very long sentences, heavy passive voice, undefined acronyms, and vague filler.
  • Solicitation-aware (v3) — the outline now uses the solicitation context it already has: tips reflect this funder's page limits, required attachments, and eligibility, and the "match THIS solicitation" panel shows in outline mode too. Each section maps (via SECTION_SAMPLES) to a worked example in the Sample Proposals Library, surfaced as a "See a worked example" link — and the link is validated with get_sample so it can never 404. In feedback mode, the review also flags any stated solicitation requirement the draft doesn't address.
Note (grounding, and a real bug it fixed): like the Draft Critic, the Coach runs _verify_evidence on every "covered" claim — it must quote the draft, or it is dropped. The membership check collapses whitespace on both sides first (" ".join(s.split())). This matters because a pasted, hard-wrapped draft has a newline where Gemini's quote has a space (e.g. "health,\nand" vs "health, and"); a raw substring match would fail and wrongly mark a complete section "NOT FOUND." Collapsing whitespace is what stopped present sections from being reported as missing.

5.5 Compliance Sentinel (which approvals do I need?)

The Compliance Sentinel answers a question every new PI has: which research-compliance approvals does this proposal need? Code lives in backend/services/compliance_sentinel.py; the UI is ComplianceSentinelModal.jsx. It has no AI at all — every decision is a plain code rule, and the plain-English explanations are deterministic templates. (The chat assistant can explain compliance topics; this tool decides requirements deterministically.)

The PI answers a short yes/no questionnaire — does the project involve human subjects, animals, a financial interest, foreign collaboration, or export-controlled technology? Combined with the proposal's sponsor, five code rules produce a checklist:

  • IRB (human subjects), IACUC (animals), COI (a financial interest — or automatically when the sponsor is NIH/PHS), RCR training (required when the sponsor is NSF or NIH), and Export Control / Research Security (export-controlled tech, or a "review" flag for foreign collaboration).

Each item comes back marked required, not required, or review, with a plain-English "why," a timing note, and a resolved link to the matching ORA form in the knowledge base (and no link at all if the form can't be resolved, so there's never a dead link). An unanswered question defaults to "review" — it never invents a requirement.

The PI can save their answers (stored in submissions.compliance_json) and click "Add required to my proposal," which creates a checklist task for each required item (review/not-required items are advisory and are skipped). That step is idempotent — running it twice won't duplicate tasks. Endpoints: GET /api/compliance/questions, POST /api/compliance/assess, and the per-proposal .../compliance save and .../compliance/tasks add.

5.6 Draft Critic (check the whole package before you submit)

The Drafting Coach grades the writing of one section. The Draft Critic checks the whole package's completeness and compliance: is it under the page limit? are all required sections and solicitation-required attachments present? is the budget under the cap? Code lives in backend/services/draft_critic.py.

You upload your draft to POST /api/me/submissions/{submission_id}/critique. Just like the solicitation upload, the draft PDF is read into memory only and never stored — it is parsed, checked, and discarded when the request ends.

Where the "rules" come from: reconstructed solicitation context

Draft Critic needs to know the rules to check against. It gets them by rebuilding them from the proposal record, using reconstruct_solicitation_context. This function reads the submissions.notes lines that Solicitation Ingestion wrote earlier — the budget cap, the page limits, the required attachments — and turns them back into a small rules dictionary. This is the "shared notebook" idea in action: Solicitation Ingestion wrote the rules into notes; Draft Critic reads them back out. The tools never talk; they just share the record.

The two layers: deterministic core (boss) + advisory AI (helper)

The single entry point is critique_pdf. It produces two layers, and it is important which one is in charge.

Layer 1 — the deterministic core, which is AUTHORITATIVE (the boss). "Deterministic" means it uses plain, predictable code with no AI — so there is zero chance of hallucination. These checks run in order:

  • Sanity check — does this PDF look like a sponsor solicitation instead of a proposal? (catches the common mistake of uploading the wrong file).
  • Page count vs the document-wide limit.
  • Required attachments — is each one present as a real heading in the draft? It uses "header-aware" detection — a section counts as present only if its name appears at the start of a line, not buried in a sentence — which avoids false "found it!" matches.
  • Standard sponsor sections — the usual NSF/NIH/DoD skeleton, with anything already flagged above suppressed so you don't see it twice.
  • Project Summary word count, per-section page limits for DMP/biosketch etc., and budget vs cap (which understands "$2.5 million" as well as plain numbers).

All these results roll up into a one-line verdict banner: "Critical issues" (red) if anything failed, "Needs review" / "Minor issues" (amber) for warnings, or "Looks ready" (green) if all clear. Because this layer is pure code, if it says "Biographical Sketch: MISSING," the user can trust that string really is absent from the document.

Layer 2 — the advisory AI review (the helper). After the deterministic checks finish, an optional Gemini review is added under a separate key called ai_review (turned on by the include_ai=True argument). It gives a plain-English summary, judgment calls the rules can't make, and rewrite suggestions. It runs against a strict instruction sheet which tells it to ground every claim in the draft and quote its evidence.

Crucially, this AI layer is advisory only — it never changes the verdict, checks, or counts. And it gets the same evidence-verification treatment: _verify_evidence drops any finding whose quote is not actually a substring of the draft, and drops any "the draft does X" claim that came with no quote at all. So if the AI invents a quote or a feature, that whole finding is thrown away before you ever see it. If the AI is unavailable or errors, ai_review is simply None and the deterministic output is unchanged.

Note: A clean way to remember it: the rules engine is the judge whose verdict is final; the AI is a colleague offering a second opinion in a clearly-labeled advisory box. The colleague can be ignored, can be wrong, or can be silent — and the judge's verdict never moves.

5.7 Deadline Watcher (reminder emails that never double-send)

This agent emails faculty when one of their in-flight proposals is approaching its deadline. Code lives in backend/services/deadline_watcher.py.

When reminders fire

Reminders go out at exactly five points before the deadline: THRESHOLD_DAYS = (14, 7, 3, 1, 0) — so 14 days out, 7 days out, 3 days out, 1 day out, and on the due day (0). These are exact matches: a proposal that is 8 days out fires nothing; it waits until it hits 7. There are no negative buckets — once a deadline passes, the watcher stops nagging.

Idempotency: it won't send the same reminder twice

"Idempotency" is a fancy word for a simple promise: running the reminder check twice produces the same result as running it once — you never get a duplicate email. Every time an email is successfully sent, the watcher writes one row into the deadline_reminder_log table recording which proposal (submission_id) and which threshold (threshold_days) it was for. On the next run, find_due_reminders skips any proposal-and-threshold pair that already has a log row. So you can re-run the check safely, the daily cron can fire twice, and a network retry can happen — and the faculty member still gets each reminder exactly once.

The log row is written only after a successful send. If the email fails, no row is written, so the next run will retry. And the log row cascade-deletes if the proposal is deleted — so a user who withdraws and recreates a proposal legitimately gets the full reminder schedule again.

The AI part: a personalized email, with a hard safety net

The orchestrator send_due_reminders tries to write a warmer, personalized email body using Gemini via compose_reminder_email_ai. The AI can do two things: write 2-4 friendly sentences whose urgency scales with how many days are left, and reorder the open checklist tasks by what to do first. Its instruction sheet forbids it from inventing any specifics — no fake dates, dollar amounts, or names.

The anti-hallucination guard here: the AI's reordered task list is accepted only if it is a valid rearrangement of exactly the real open tasks. If the AI dropped, renamed, or invented a task, the code throws away the AI ordering and keeps the original — so a made-up task title can never appear in the email.

And there is a hard fallback so a reminder always sends. If the AI is slow, down, or fails for any reason, compose_reminder_email_ai returns None, and the code falls back to the deterministic template compose_reminder_email. The email subject is always deterministic (for a consistent inbox), and only the body prose is ever AI-written.

What triggers it

The watcher runs when something POSTs to /api/internal/deadlines/check. This endpoint is protected by a shared secret in the X-Research-Secret request header, matched against the RESEARCH_SECRET environment variable — so random internet traffic can't trigger it. In production, Google Cloud Scheduler calls this endpoint automatically once a day (at 7am Eastern). The endpoint just opens a database session and calls send_due_reminders(db), then returns a small JSON summary (how many scanned, sent, failed).

5.8 The Guided Proposal Pathway (/my-proposals)

This is the page where a faculty member sees and manages their proposals. The UI code is frontend/src/components/MyProposals.jsx. For an inexperienced PI, a flat toolbar of seven tools is intimidating — so this screen reshapes the same tools into a staged pathway (no new tool logic, just presentation and orchestration).

What the user sees:

  • A list of their proposal cards, each showing the title, sponsor, status pill (Active / Submitted / Withdrawn), and a countdown to the deadline that turns urgent when 7 or fewer days remain and "overdue" when it's past.
  • The summary also shows the internal ORA routing deadline — about 5 business days before the sponsor's date (weekends skipped), computed by proposals_service.internal_routing_deadline and delivered as the internal_deadline field. This is the real institutional cutoff a first-timer should plan backward from.

The three stages and the "next step"

Inside a proposal, the tools are grouped into three stages — Set up → Build → Review & submit:

  • Set up — the Solicitation (a status, not a step: it can only be attached at creation).
  • Build — Budget Helper and Drafting Coach.
  • Review & submit — Compliance Sentinel and (only when the proposal carries solicitation rules) Draft Critic.

Each tool carries a ✓ status badge driven by serializer flags: has_budget, has_sections (true when sections_json is saved), has_compliance, and a hasSolicitation check on the notes lines (the Critic is stateless, so it has no badge). A pure function, nextStep(submission), picks the single recommended action in order — budget → coach → compliance → critique → done — and only that one button gets the accent color; the rest share one calm style. A plain-language "What to do next" card explains the current step and why (shown for active proposals only). The less-common actions — Delete, and Add to Google Calendar — live behind a ⋯ overflow menu so they don't compete with the recommended step.

The checklist, templates, and per-task guidance

Each proposal owns a checklist of tasks (a check circle, the title, an optional description, and a "N days before deadline" hint; clicking the circle toggles done). When a proposal is created its checklist isn't blank — it's seeded from a template in backend/services/proposal_templates.py (a generic checklist for any sponsor, plus NSF- and NIH-specific add-ons; get_template combines them). There's a special NSF Education-program-only task that is appended only when the program text actually looks like an NSF education program — so a plain NSF science proposal doesn't get an irrelevant education checklist.

Each task also carries plain-English help from backend/services/task_guidance.py: guidance_for(title) keyword-matches the task title to a how-to (and sometimes a short sample), attached as guidance in the task serializer. There's a per-task "Add to calendar" button (a Google Calendar link), plus a per-user .ics deadline feed.

The proposal/task API endpoints

The tracker talks to a small set of endpoints in main.py: list (GET /api/me/submissions), create (which seeds the template), get one, update, delete (which cascade-deletes its tasks), plus add / update / delete individual tasks, and a template list for the create dropdown. Every one of these is gated so a user can only touch their own proposals.

5.9 The Forms catalog (/forms)

The Forms catalog is a browsable list of ORA's forms, templates, checklists, and memos — so a faculty member can open the right PDF in one click without asking the chatbot (which would burn an AI call and risk a made-up link). Code lives in backend/services/forms_catalog.py.

Where the catalog data comes from: it is a pure static read of a bundled file — no database, no AI, no network. At import time it reads kb_structured/_all_documents.jsonl once, keeps only the "form-like" documents (those whose subcategory mentions words like form, template, docusign, checklist, memo, sample), and for each one derives extra tags it can be filtered by:

  • sponsor — guessed by keyword-matching the title and content; "Internal" if nothing external matches.
  • roles — guessed from the document's category (e.g. pre-award forms serve "PI" and "Staff"; a form can have several roles).

Each catalog entry includes the clickable form link (url), a "View on morgan.edu" source link (source_url), a short summary, and the sponsor/role tags. The endpoint GET /api/forms calls list_forms with optional category, sponsor, and role filters. Because everything is read straight from the bundled knowledge base, every link is real — nothing here is invented by an AI.

5.10 Opportunity Finder + Fit Advisor (/opportunities)

The Opportunity Finder closes the gap before a PI even has a solicitation: they describe their research in plain English and get a ranked list of live, open federal opportunities, each with a grounded fit explanation, an eligibility verdict, and a mechanism note. It's a standalone page (navbar "Find Funding"), not a per-proposal tool. Logic in backend/services/opportunity_finder.py; endpoint POST /api/opportunities/search (the search also pulls in the user's saved interest memories to enrich the query).

  • Source = Grants.gov public API (fixed host api.grants.gov, no key, covers all 26 federal agencies). It calls search2 (keyword → open opportunity ids/basics) then fetchOpportunity per top result for the rich synopsis (applicant types, eligibility description, synopsis text, award ceiling, close date, contact). It returns an empty list — never fabricated results — on any API failure.
  • Eligibility = a deterministic STOP-gate (eligibility_gate). Morgan State's profile is baked in (public institution of higher education, HBCU, MSI — just like the F&A rates are baked in) and matched against each opportunity's applicant-type descriptions → eligible / unrestricted / see_text / ineligible. AI is advisory only.
  • Ranking = AI advisory, grounded (rank_and_explain). Gemini re-ranks and writes a 1–2 sentence fit note, each backed by a verbatim quote that is verified against the synopsis text (unquotable quotes are dropped — the same whitespace-collapse pattern as the Coach). The model gets max_output_tokens=4096 (a smaller budget truncated the ranking JSON and silently fell back). When Gemini is down it falls back to the API's relevance order with no AI prose.

Each result also shows the internal ORA routing deadline (5 business days before close) and a mechanism note for recognized NIH activity codes (R01/R21/K-series, etc.). Nothing is persisted — the page only reads the user's interest memories to enrich the query; the live API results live in the browser. "Start a proposal from this" routes the opportunity's URL into the Solicitation upload modal (via router state and the modal's initialUrl prop), dropping the PI straight into the guided pathway. Tests: tests/test_opportunity_finder.py (unit) and tests/test_opportunities_api_e2e.py (route/auth).

5.11 Sample Proposals Library (/sample-proposals)

So an inexperienced PI can see what a fundable proposal actually reads like, there's a browse surface at /sample-proposals (navbar "Samples," styled like the Forms catalog). It's a hybrid of two entry kinds, ranked authored-first and filtered by a category chip (NSF / NIH / Foundations / Early-career). Data and helpers are a static constant in backend/services/sample_proposals.py — no database, no LLM, no network.

  • Type "pdf" — original samples we authored ourselves and host for download (zero copyright exposure; each PDF carries a "not a real submitted proposal" disclaimer). Three today: a full NSF planning proposal (EJ-IDSS), an NIH Specific Aims + Research Strategy excerpt, and an annotated Budget Justification.
  • Type "link" — vetted links out to authoritative public libraries (NSF's official sample, NIH/NIAID, USF/UAF/Northwestern/UChicago/Carleton-SERC, Open Grants, Morgan ORA). We never rehost these — they're copyrighted and often index pages, not single files.

Endpoints: public GET /api/sample-proposals?category= (listing) and public GET /api/sample-proposals/{id}/download (a FileResponse; 404 for an unknown id, a "link" entry, or a missing file). The page sits behind RequireAuth like /forms, but the endpoints are auth-free because the content is all public.

Note (how the PDFs ship): the authored PDFs are generated by scripts/make_sample_proposals.py (reportlab, dev-only) into backend/sample_proposals/ and are committed so they ride along in the Docker image — .gitignore has a !backend/sample_proposals/*.pdf negation past the blanket *.pdf rule, without which the download would 404 in production. The backend needs no runtime PDF dependency — it only serves the pre-generated files.

5.13 How your profile feeds the tools

A user's profile fields — Department, Title, Role, and research Interests — aren't just decoration; they quietly flow into the system that personalizes answers. The bridge is mirror_profile_to_memories, called from the profile-save path.

"Mirroring" means copying the profile values into the user_memories table as memory facts: department becomes a memory of type department, the role becomes a memory of type role, and each interest becomes its own interest memory. (Title is intentionally not mirrored — there's no matching memory type.) A profile value overwrites any conflicting fact the system auto-learned earlier — the profile wins.

Why bother? Because the chatbot already builds its context for each answer by reading the user_memories table through build_memory_context, and the Opportunity Finder reads the same interest memories to enrich a funding search. By mirroring the profile into that one table, the values get picked up automatically — with no code change needed in the agent, chat path, or finder. It's the shared-notebook pattern again: instead of teaching each feature to read profile fields directly, the profile writes itself into the place they already look. So a faculty member who sets their role to "PI" and department to "Chemistry" will get answers tailored to a Chemistry PI, simply because those facts now live in the memory the tools already consult.
