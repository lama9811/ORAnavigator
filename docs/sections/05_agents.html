5. The AI Agents & Proposals / Forms

This section explains the part of ORA Navigator that helps a faculty member actually build and submit a grant proposal — not just chat about grants. There are three small AI helpers (called "agents"), plus the screens and database tables they read and write. Everything here is wired up so a wrong guess by the AI can never quietly cause harm: a human always reviews before anything is committed, and the AI's output is double-checked against the real document.

5.1 The big idea: agents that coordinate through a shared record

An "agent" here just means a focused piece of software that does one job and can call the AI model (Gemini) when it needs to. ORA Navigator has three agents:

  • Solicitation Ingestion — reads a sponsor's grant announcement PDF and pulls out the key facts.
  • Draft Critic — checks a faculty member's draft proposal PDF against the rules.
  • Deadline Watcher — emails faculty as a deadline gets close.
Note: There used to be a fourth agent called "Sponsor Fit-Finder" (it suggested funding sources). It was removed on 2026-06-01. The funding knowledge-base documents were kept, so the chatbot can still answer funding questions — but the agent itself is gone. So today there are exactly three agents.

The most important design point: the agents never call each other directly. Instead, they all read and write the same proposal record in the database. Think of it like a shared notebook on a desk. Solicitation Ingestion writes the proposal's facts into the notebook. Later, Draft Critic opens the same notebook to learn the rules. Later still, Deadline Watcher opens it to see when things are due. None of them phones the others — they just share the notebook (the submissions and submission_tasks tables in the database).

All three are real LLM-backed agents — they genuinely call the Gemini AI model. But every one of them is protected by evidence verification, not just polite instructions. "Evidence verification" means: after the AI says something, the code mechanically checks whether the AI's claim is actually backed by a real quote from the source document. If the AI invents a fact, the code catches it and throws it away. This is the difference between asking the AI to behave ("please don't make things up") and forcing it to ("if you can't quote it, we delete it").

The shared Gemini client

All three agents reach Gemini through one shared helper, backend/services/gemini_client.py — "one reusable phone line to Gemini 2.5 Flash." It does three helpful things:

  • Builds the connection once and reuses it — no slow setup on every call.
  • Fails fast and safely. If the AI can't be reached at all (for example, running offline or in a test), every call returns None immediately instead of hanging. This is what lets each agent quietly fall back to its non-AI behavior.
  • Never crashes. The two public functions generate_text and generate_json always return either a clean result or None — they never raise an error. generate_json also strips out any stray markdown code-fences and tolerates odd control characters that PDFs sometimes leak.

The model name is hard-coded to "gemini-2.5-flash" because the older gemini-2.0-flash name returns a "not found" error in this Google Cloud project.

5.2 Agent 1 — Solicitation Ingestion (PDF in, structured proposal out)

A "solicitation" (also called an FOA, "Funding Opportunity Announcement") is the long PDF a sponsor like NSF or NIH publishes to describe a grant: who can apply, how much money, the deadline, page limits, what attachments are required. Reading it carefully is tedious and error-prone. This agent does the first pass for you. Code lives in backend/services/solicitation_extractor.py.

Step by step, what happens to the PDF

  1. You upload the PDF. The browser sends it to the endpoint POST /api/me/submissions/from-solicitation. The file must be a PDF and at most 25 MB.
  2. The PDF is read into memory only. The endpoint does pdf_bytes = await file.read() — the bytes live in RAM for the length of the request. The PDF file is never saved to disk and never stored in any database table. Once the request finishes, the bytes are gone.
  3. pdfplumber turns the PDF into plain text. extract_text_from_pdf opens the PDF with the pdfplumber library and pulls the text out of each page. It also strips illegal control characters that pdfplumber sometimes emits — those would otherwise break the next step.
  4. Gemini extracts the facts as JSON. The text (up to 250,000 characters — a big window so facts buried deep in long FOAs aren't missed) is sent to Gemini with a very strict instruction sheet called _EXTRACT_SYSTEM. Gemini must return a JSON object with exactly these fields: sponsor, program_id, program_name, deadline, page_limits, required_attachments, eligibility, budget_cap, submission_portal, and source_quotes.

The source_quotes field is the clever part. For every value Gemini fills in, it must also paste the exact verbatim sentence from the PDF that proves that value. If Gemini can't find a real supporting quote, the rules tell it to return null ("no quote, no value"). The instruction sheet also forces other safety rules: never guess a deadline, use the most restrictive number when several are given (smallest budget cap, smallest page limit, so an applicant is never told they have more room than they do), and budget_cap means the maximum per award, not the total program budget.

The anti-hallucination check: flag, don't drop

Telling the AI "always quote your source" is not enough — an AI can paste a fake quote. So after Gemini answers, a deterministic (non-AI) function _verify_source_quotes checks each filled field: it normalizes the text (lowercase, drop bullet symbols, collapse spaces) and asks "is this quote actually a substring of the real PDF text?" If a field has a value but no real quote behind it, that field name is added to a list called unverified_fields.

The key policy is "flag, don't drop": the suspicious value is not deleted. It is kept but marked, and the UI shows it in red so the human reviewing it knows to double-check that one. This is safe because a human reviews everything before it becomes a real proposal.

Note: A wrong deadline can cost a faculty member an entire grant. That is exactly why this agent never creates a proposal by itself. Step 1 only returns the extracted facts for review. Nothing is saved yet.

Step two: the human confirms, and a Submission is born

After the user reviews and edits the extracted fields on screen, the browser calls a second, separate endpoint, POST /api/me/submissions/from-solicitation/confirm. This two-step design means a wrong AI guess can never auto-commit — the user must explicitly confirm.

Confirm calls create_submission_from_solicitation, which is where the extracted facts finally get saved to the database:

  • A new row is inserted into the submissions table: the title, sponsor, parsed deadline, status="active", and a notes text blob.
  • The notes column carries the leftover facts in a structured, machine-readable format — lines like "Budget cap: $600,000", "Page limits: project_description: 15p, ...", and "Required attachments: ...". This matters later: Draft Critic reads these exact lines back out.
  • The checklist tasks are inserted into the submission_tasks table: first the sponsor's standard template tasks, then one extra task per required attachment that the template didn't already cover (titled "Prepare required attachment: X").

5.3 Agent 2 — Draft Critic (check your draft before you submit)

Once you've written your actual proposal, Draft Critic gives it a pre-submission once-over: is it under the page limit? are all required attachments present? is the budget under the cap? Code lives in backend/services/draft_critic.py.

You upload your draft to POST /api/me/submissions/{submission_id}/critique. Just like the solicitation upload, the draft PDF is read into memory only and never stored — it is parsed, checked, and discarded when the request ends.

Where the "rules" come from: reconstructed solicitation context

Draft Critic needs to know the rules to check against. It gets them by rebuilding them from the proposal record, using reconstruct_solicitation_context. This function reads the submissions.notes lines that Solicitation Ingestion wrote earlier — the budget cap, the page limits, the required attachments — and turns them back into a small rules dictionary. This is the "shared notebook" idea in action: Agent 1 wrote the rules into notes; Agent 2 reads them back out. The two agents never talk; they just share the record.

The two layers: deterministic core (boss) + advisory AI (helper)

The single entry point is critique_pdf. It produces two layers, and it is important which one is in charge.

Layer 1 — the deterministic core, which is AUTHORITATIVE (the boss). "Deterministic" means it uses plain, predictable code with no AI — so there is zero chance of hallucination. These checks run in order:

  • Sanity check — does this PDF look like a sponsor solicitation instead of a proposal? (catches the common mistake of uploading the wrong file).
  • Page count vs the document-wide limit.
  • Required attachments — is each one present as a real heading in the draft? It uses "header-aware" detection — a section counts as present only if its name appears at the start of a line, not buried in a sentence — which avoids false "found it!" matches.
  • Standard sponsor sections — the usual NSF/NIH/DoD skeleton, with anything already flagged above suppressed so you don't see it twice.
  • Project Summary word count, per-section page limits for DMP/biosketch etc., and budget vs cap (which understands "$2.5 million" as well as plain numbers).

All these results roll up into a one-line verdict banner: "Critical issues" (red) if anything failed, "Needs review" / "Minor issues" (amber) for warnings, or "Looks ready" (green) if all clear. Because this layer is pure code, if it says "Biographical Sketch: MISSING," the user can trust that string really is absent from the document.

Layer 2 — the advisory AI review (the helper). Since 2026-06-03, after the deterministic checks finish, an optional Gemini review is added under a separate key called ai_review (turned on by the include_ai=True argument). It gives a plain-English summary, judgment calls the rules can't make, and rewrite suggestions. It runs against a strict instruction sheet which tells it to ground every claim in the draft and quote its evidence.

Crucially, this AI layer is advisory only — it never changes the verdict, checks, or counts. And it gets the same evidence-verification treatment: _verify_evidence drops any finding whose quote is not actually a substring of the draft, and drops any "the draft does X" claim that came with no quote at all. So if the AI invents a quote or a feature, that whole finding is thrown away before you ever see it. If the AI is unavailable or errors, ai_review is simply None and the deterministic output is unchanged.

Note: A clean way to remember it: the rules engine is the judge whose verdict is final; the AI is a colleague offering a second opinion in a clearly-labeled advisory box. The colleague can be ignored, can be wrong, or can be silent — and the judge's verdict never moves.

5.4 Agent 3 — Deadline Watcher (reminder emails that never double-send)

This agent emails faculty when one of their in-flight proposals is approaching its deadline. Code lives in backend/services/deadline_watcher.py.

When reminders fire

Reminders go out at exactly five points before the deadline: THRESHOLD_DAYS = (14, 7, 3, 1, 0) — so 14 days out, 7 days out, 3 days out, 1 day out, and on the due day (0). These are exact matches: a proposal that is 8 days out fires nothing; it waits until it hits 7. There are no negative buckets — once a deadline passes, the watcher stops nagging.

Idempotency: it won't send the same reminder twice

"Idempotency" is a fancy word for a simple promise: running the reminder check twice produces the same result as running it once — you never get a duplicate email. Every time an email is successfully sent, the watcher writes one row into the deadline_reminder_log table recording which proposal (submission_id) and which threshold (threshold_days) it was for. On the next run, find_due_reminders skips any proposal-and-threshold pair that already has a log row. So you can re-run the check safely, the daily cron can fire twice, and a network retry can happen — and the faculty member still gets each reminder exactly once.

The log row is written only after a successful send. If the email fails, no row is written, so the next run will retry. And the log row cascade-deletes if the proposal is deleted — so a user who withdraws and recreates a proposal legitimately gets the full reminder schedule again.

The AI part: a personalized email, with a hard safety net

The orchestrator send_due_reminders tries to write a warmer, personalized email body using Gemini via compose_reminder_email_ai. The AI can do two things: write 2-4 friendly sentences whose urgency scales with how many days are left, and reorder the open checklist tasks by what to do first. Its instruction sheet forbids it from inventing any specifics — no fake dates, dollar amounts, or names.

The anti-hallucination guard here: the AI's reordered task list is accepted only if it is a valid rearrangement of exactly the real open tasks. If the AI dropped, renamed, or invented a task, the code throws away the AI ordering and keeps the original — so a made-up task title can never appear in the email.

And there is a hard fallback so a reminder always sends. If the AI is slow, down, or fails for any reason, compose_reminder_email_ai returns None, and the code falls back to the deterministic template compose_reminder_email. The email subject is always deterministic (for a consistent inbox), and only the body prose is ever AI-written.

What triggers it

The watcher runs when something POSTs to /api/internal/deadlines/check. This endpoint is protected by a shared secret in the X-Research-Secret request header, matched against the RESEARCH_SECRET environment variable — so random internet traffic can't trigger it. In production, Google Cloud Scheduler calls this endpoint automatically once a day (at 7am Eastern). The endpoint just opens a database session and calls send_due_reminders(db), then returns a small JSON summary (how many scanned, sent, failed).

5.5 The Proposals tracker screen (/my-proposals)

This is the page where a faculty member sees and manages their proposals. The UI code is frontend/src/components/MyProposals.jsx.

What the user sees:

  • A list of their proposal cards, each showing the title, sponsor, status pill (Active / Submitted / Withdrawn), and a countdown to the deadline that turns urgent when 7 or fewer days remain and "overdue" when it's past.
  • Inside a proposal, the checklist of tasks. Each TaskRow shows a check circle, the task title, an optional description, and a "N days before deadline" hint. Clicking the circle toggles the task between pending and done.
  • Buttons to upload a solicitation PDF (opening SolicitationUploadModal) and to critique a draft (DraftCritiqueModal). The "Critique Draft" button only appears for proposals that actually carry solicitation rules — the frontend checks the same notes lines the backend wrote.

Proposal templates (the seeded checklist)

When a proposal is created, its checklist isn't blank — it's seeded from a template so the user lands on a populated to-do list. Templates live in backend/services/proposal_templates.py. There is a _GENERIC_CHECKLIST used for any sponsor, plus NSF-specific and NIH-specific add-on tasks. get_template returns the right combined list for the sponsor. There's also a special NSF Education-program-only task that is appended only when the program text actually looks like an NSF education program — so a plain NSF science proposal doesn't get an irrelevant education checklist.

The kb_doc_id "Open form" link — and an honest caveat

Each template task can carry a kb_doc_id — a pointer to the matching form/template in the knowledge base, the idea being the UI could show an "Open form" link right on the task row. For example, "Build the proposal budget" points to form_budget_justification_template. This value is saved on the task row (column submission_tasks.kb_doc_id) and is sent to the browser by the API.

Note (verified against the code): The frontend does not yet use kb_doc_id. The TaskRow component renders only the title, description, and the days-before-deadline hint — there is no "Open form" link anywhere in the file. So the data is delivered but unused: the "Open form" button is still to be built. (Tasks that have no matching KB form — biosketch, DMP, Specific Aims, etc. — correctly leave kb_doc_id as None so no dead link could ever be created.)

The proposal/task API endpoints

The tracker talks to a small set of endpoints in main.py: list (GET /api/me/submissions), create (which seeds the template), get one, update, delete (which cascade-deletes its tasks), plus add / update / delete individual tasks, and a template list for the create dropdown. Every one of these is gated so a user can only touch their own proposals.

5.6 The Forms catalog (/forms)

The Forms catalog is a browsable list of ORA's forms, templates, checklists, and memos — so a faculty member can open the right PDF in one click without asking the chatbot (which would burn an AI call and risk a made-up link). Code lives in backend/services/forms_catalog.py.

Where the catalog data comes from: it is a pure static read of a bundled file — no database, no AI, no network. At import time it reads kb_structured/_all_documents.jsonl once, keeps only the "form-like" documents (those whose subcategory mentions words like form, template, docusign, checklist, memo, sample), and for each one derives extra tags it can be filtered by:

  • sponsor — guessed by keyword-matching the title and content; "Internal" if nothing external matches.
  • roles — guessed from the document's category (e.g. pre-award forms serve "PI" and "Staff"; a form can have several roles).

Each catalog entry includes the clickable form link (url), a "View on morgan.edu" source link (source_url), a short summary, and the sponsor/role tags. The endpoint GET /api/forms calls list_forms with optional category, sponsor, and role filters. Because everything is read straight from the bundled knowledge base, every link is real — nothing here is invented by an AI.

5.8 How your profile feeds the agents

A user's profile fields — Department, Title, Role, and research Interests — aren't just decoration; they quietly flow into the system that personalizes answers. The bridge is mirror_profile_to_memories, called from the profile-save path.

"Mirroring" means copying the profile values into the user_memories table as memory facts: department becomes a memory of type department, the role becomes a memory of type role, and each interest becomes its own interest memory. (Title is intentionally not mirrored — there's no matching memory type.) A profile value overwrites any conflicting fact the system auto-learned earlier — the profile wins.

Why bother? Because the chatbot already builds its context for each answer by reading the user_memories table through build_memory_context. By mirroring the profile into that same table, the profile values get picked up automatically — with no code change needed in the agent or chat path. It's the shared-notebook pattern again: instead of teaching the chatbot to read profile fields directly, the profile writes itself into the place the chatbot already looks. So a faculty member who sets their role to "PI" and department to "Chemistry" will get answers tailored to a Chemistry PI, simply because those facts now live in the memory the agent already consults.
