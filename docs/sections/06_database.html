6. The Database, Caching & Authentication

This section explains three "behind the scenes" systems that ORA Navigator depends on: the database (where all the durable information is stored), the cache (a memory of recent answers so the app doesn't redo expensive work), and authentication (how the app knows who you are and whether you are allowed in).

6.1 What a database is, and the tables ORA Navigator uses

A database is an organized collection of information stored on a server so it survives restarts. A table inside a database is like one spreadsheet: it has named columns (the kinds of fact it stores) and rows (one record each). ORA Navigator uses a MySQL 8.4 database (Google Cloud SQL instance oranavigator-db). The tables are described in Python using a tool called SQLAlchemy ORM ("ORM" = Object-Relational Mapping, which means each table is written as a Python class and each column as a class attribute). All these classes live in backend/models.py.

Note: A few columns are typed Text but actually hold JSON text (a list or object turned into a string with Python's json.dumps, and read back with json.loads). So when you see citations, questions, embedding, or sources below, picture a piece of text that, when decoded, becomes a list.

Table: users

Purpose: one row per person who has an account. It holds their login credentials, their role (regular user or admin), their profile, and several "memory" control flags.

idA unique number for the user (the "primary key"). Other tables point back to this number.
emailThe user's email address. Must be unique; used to log in.
password_hashA scrambled version of the password (never the real password).
roleEither "user" (default) or "admin". Admins can see admin-only screens.
nameThe user's display name. Optional.
profile_picture / profile_picture_dataA path/URL to a profile image, or the actual image stored inline as base64 text.
email_verifiedTrue/False: has the user clicked the email verification link yet?
verification_token / reset_token / reset_token_expiresOne-time codes for email verification and "forgot my password" resets, plus when the reset code stops working.
created_atWhen the account was created.
last_chat_atWhen the user last chatted. The "idle-sweep" memory job uses this to extract facts 5-10 minutes after someone goes quiet.
memory_pausedTrue/False: if True, the app stops recalling and extracting personal memory for this user (a global privacy switch).
department / titleThe user's department and job title. Optional; help the agent tailor answers.
primary_roleTheir research-admin role. Allowed values are validated in code: PI, Co-PI, Research Staff, Department Admin, Faculty, Postdoc, Student.

Table: chat_history

Purpose: one row per chat turn (a question and its answer). It also stores the Sources shown with the answer, plus extra fields used by the long-term memory system.

idUnique number for this chat turn.
user_idWhich user this turn belongs to.
session_idWhich conversation thread this turn is part of (defaults to "default").
user_query / bot_responseThe exact text the user typed and the exact answer the bot gave.
citationsJSON list of {title, url} Sources for this answer. Saving them here means the "Sources" box survives cache hits, page refreshes, and history reloads.
timestampWhen the turn happened.
session_summary / summary_through_idA rolling plain-text summary of the conversation so far, and the last chat id the summary already covers.
embedding / embedding_modelA JSON list of numbers representing this turn's meaning (used for semantic recall across sessions), plus the model that produced it.
topic_labelA short label for the turn's topic.

Table: feedback

Purpose: records when a user rates an answer (thumbs up/down or a report), so the team can improve the bot.

id, user_id, session_idIdentity of the record, who gave it, and which conversation.
message_textThe bot message the feedback is about.
feedback_typeOne of "helpful", "not_helpful", or "report".
report_detailsOptional extra text the user wrote when reporting.
timestampWhen the feedback was submitted.

Table: support_tickets

Purpose: bug reports, feature requests, and questions submitted by users, with status tracking and an admin response area.

id, user_idTicket number and who opened it.
subject / descriptionShort title and full description of the issue.
categoryOne of "bug", "feature", "question", "other".
attachment_data / attachment_nameAn optional attached file stored as base64 text (up to ~12MB), and its name.
statusOne of "open", "in_progress", "resolved", "closed" (defaults to "open").
priorityOne of "low", "normal", "high", "urgent" (defaults to "normal").
admin_notes, resolved_by, resolved_atInternal notes, the admin who resolved it, and when.
created_at / updated_atWhen it was opened and last changed.

Table: user_memories

Purpose: long-term facts about each user, distilled from their conversations by a nightly job. These are short one-line facts (not raw chat), used to personalize answers.

id, user_idUnique number and which user this fact is about.
memory_typeThe kind of fact: role, department, active_grant, irb_protocol, iacuc_protocol, sponsor, interest, preference, goal, or context.
contentThe fact itself, in plain words (e.g. "Works in the Biology department").
created_at / updated_atWhen the fact was first stored and last changed.
embedding / embedding_modelA JSON list of 256 numbers (a meaning-vector) so the app can find the most relevant facts by similarity, plus the model that produced it.
pausedTrue/False: if True this single fact is skipped during recall (a per-fact privacy switch, separate from users.memory_paused).

Table: user_suggested_questions

Purpose: the personalized "you might want to ask…" prompts shown on the home screen, precomputed per user so the page loads instantly.

user_idWhich user these suggestions belong to. This is also the primary key (one row per user).
questionsA JSON list of suggested question strings.
generated_atWhen the suggestions were last generated.
source_signatureA fingerprint like "{max_chat_id}:{max_memory_updated_epoch}" used to skip regeneration when nothing has changed.
sourceEither "personalized" or "default" (a generic cold-start sample).

Table: failed_queries

Purpose: records questions the bot couldn't answer well (knowledge-base "misses"), so an auto-research process can later find and fill those gaps.

idUnique number for this failed query.
user_query / bot_responseThe question the bot struggled with and what it said (often a refusal).
user_idWho asked (optional — can be empty for guests).
cluster_idA group number linking similar failed questions together.

created_atWhen it was logged.

Table: kb_suggestions

Purpose: proposed knowledge-base additions generated by the auto-research agent. An admin reviews and approves each one before it goes into the live knowledge base.

admin_notes, reviewed_by, reviewed_atReviewer's notes, who reviewed it, and when.

Table: submissions

Purpose: one row per in-progress grant proposal a user is tracking. Each submission owns an ordered checklist of tasks.

id, user_idUnique number and the owner.
title / sponsorThe proposal's name and who is being applied to (defaults to "Internal").
deadlineWhen the proposal is due (optional).
statusOne of active (default), submitted, withdrawn.
notesFree-text notes — and the structured solicitation rules (budget cap, category caps, page limits, required attachments, eligibility) that the Budget Helper, Drafting Coach, and Draft Critic read back.
budget_json / compliance_json / sections_jsonTEXT columns holding json.dumps of, respectively, the saved Budget Helper inputs, the Compliance Sentinel answers, and the Drafting-Coach section drafts. Each was added later by a self-healing migration. The proposal serializer turns these into the has_budget / has_compliance / has_sections badges that drive the Guided Pathway (Chapter 5).
created_at / updated_atWhen created and last changed.

Cascade behavior: deleting a submission automatically deletes its tasks.

Table: submission_tasks

Purpose: a single checklist item under a submission (for example "Upload biosketch"). Seeded from a template when the proposal is created, then editable by the user.

id, submission_idUnique number and which proposal this task belongs to (deletes with the parent).
title / descriptionThe task name and an optional longer explanation.
kb_doc_idOptional link to the knowledge-base form/template this task matches, so the UI can show an "Open form" link.
due_offset_daysHow many days before the deadline this task should be done.
statuspending (default) or done.
notes / sort_orderFree-text notes and the position of this task in the checklist.
created_at / updated_atWhen created and last changed.

Table: deadline_reminder_log

Purpose: a bookkeeping log so the Deadline Watcher never emails the same reminder twice. Each time it sends a "deadline in N days" email, it writes one row here.

id, submission_idUnique number and which proposal the reminder was for (deletes with the parent).
threshold_daysWhich "days before deadline" bucket this was: 14, 7, 3, 1, or 0.
sent_at / sent_toWhen the email was sent and the address it went to (for audit only).

Why it matters ("idempotency"): before sending, the watcher checks whether a row already exists for that (submission_id, threshold_days) pair; if it does, it skips. So a faculty member never gets two "14 days left" emails for the same proposal.

Note: The full set of tables is exactly: users, chat_history, feedback, support_tickets, user_memories, user_suggested_questions, failed_queries, kb_suggestions, submissions, submission_tasks, and deadline_reminder_log — eleven tables in total.

6.2 How the app connects to the database (two environments)

The connection code lives in backend/db.py. It reads a single setting called DATABASE_URL from the environment. If that setting is missing, it falls back to a local SQLite file so the app still starts. For the real MySQL database, the function _make_engine builds the connection in one of two ways.

(a) On a developer's laptop: TCP + SSL to the Cloud SQL public IP

"TCP" just means a normal network connection over the internet to the database server's public IP address. MySQL 8.4 requires encryption for such connections, so the code builds an SSL context (a secure-channel object) and attaches it. It sets check_hostname = False and verify_mode = CERT_NONE — meaning "encrypt the traffic, but don't insist on verifying the server's certificate name." Without this SSL context you get a confusing Access denied error even with the correct password, because MySQL 8.4 refuses unencrypted logins.

(b) On Cloud Run (production): a Unix socket via the Cloud SQL Auth Proxy

In production the app runs inside Google Cloud Run, which talks to the database through the Cloud SQL Auth Proxy — a secure local tunnel that appears as a Unix socket (a special file on disk that two programs use to talk, instead of a network address). The DATABASE_URL contains a unix_socket= setting. The code detects it and uses SQLAlchemy's "creator" pattern: instead of letting SQLAlchemy parse the URL, it hands SQLAlchemy a small function that calls pymysql.connect(...) directly with the socket path. This bypasses URL parsing entirely (which would choke on the socket path).

Note: Why two methods? A laptop is outside Google's network, so it must reach the database over the public internet — which requires SSL. Cloud Run is inside Google's network and uses the proxy/socket, which is both faster and avoids exposing the database publicly. Every connection also sets pool_pre_ping=True and pool_recycle=3600, meaning "test a reused connection before using it, and throw away connections older than one hour" so stale connections don't cause errors.

6.3 How schema changes are applied (adding tables and columns safely)

Over time the code adds new tables and new columns. There are three mechanisms, and it's important to know which one runs automatically.

  • New tables — automatic. On startup, init_db() calls Base.metadata.create_all(bind=engine). This creates any table that is defined in models.py but doesn't yet exist. It is safe to run every time: it never touches tables that already exist.
  • New columns on existing tables — self-healed by "SELECT → ALTER" checks. create_all does not add new columns to tables that already exist. So init_db() includes hand-written checks that try to SELECT the new column; if that fails (because the column is missing), it runs an ALTER TABLE … ADD COLUMN to add it. Examples: adding chat_history.session_id, users.profile_picture_data, the email-auth columns, and chat_history.citations. This is called "self-healing" because the running app fixes its own schema on boot.
  • The big migration script — NOT run automatically. backend/migrate_db.py has a migrate() function with the full, ordered list of column adds, new-table creates, legacy-cleanup steps, and the role rename from "student" to "user". This file only runs if you execute it directly. The app does not call it on startup.
Note: Practical takeaway: a brand-new table you define in models.py appears by itself (via create_all). A brand-new column on an existing table only appears in production if it is covered by a SELECT→ALTER check in init_db() — you can't rely on migrate_db.migrate() running, because it doesn't run automatically.

6.4 Caching: remembering answers so the app doesn't redo work

A cache is a short-term memory of recent results. Producing a chat answer can take a full Gemini call (about 2-5 seconds and real cost). If the same question is asked again, the cache hands back the saved answer in milliseconds. All cache code lives in backend/cache.py, exposed through one global object query_cache. It has three layers, checked in order from fastest to smartest.

Layer L1 — in-memory cache (fastest)

L1 is a TTLCache held in the server's own memory. "TTL" means Time To Live: entries expire after a set time — here 24 hours — and it holds at most 500 entries, dropping the least-recently-used when full. It is roughly 0.001 ms fast, but it is local to one server instance and is wiped on restart.

Layer L2 — Redis cache (shared and durable)

L2 is Redis, a separate fast database used purely for caching. It is shared across all server instances and survives restarts, with a 7-day expiry. Its connection string comes from the REDIS_URL environment variable (the cloud secret ora-redis-url). All keys are namespaced with an oranavigator: prefix. If Redis can't be reached, the code logs a warning and simply runs without L2 — so the app keeps working.

Layer "Semantic" — meaning-based matching

The third layer catches questions that are worded differently but mean the same thing. It turns each question into an embedding — a list of numbers representing meaning — using Google's text-embedding-004 model at 256 dimensions. To compare two questions it computes cosine similarity (a score from 0 to 1, where 1 means "same meaning"). If the best match scores at least 0.95, it reuses that cached answer. The high 0.95 bar is deliberate: it only allows near-identical matches, to avoid mixing up different topics. Example: "what is the F&A rate" matches "what indirect cost rate does Morgan State charge". It keeps at most 100 such entries in memory and also persists them to Redis so they survive restarts.

The lookup order: check L1 (exact match) → check L2/Redis (exact match) → check Semantic (similar meaning) → otherwise it's a "miss" and the AI is asked. When the AI answers, it is written to L1, L2, and Semantic together. A separate parallel store keeps the answer's Sources list under a cit:-prefixed key so a cache hit can re-show its Sources.

What is deliberately NEVER cached

The function _should_cache blocks caching for:

  • Very short questions — under 15 characters, so "hi" and "hello" aren't cached.
  • Personal-recall questions. The cache key is built only from the question text and model — it has no user_id. So if a personalized answer were cached, it could leak one user's facts to another user who asked the same words. To prevent that, any question containing words like "my grant", "my deadline", "about me", "remember", "for me" is skipped, and the code also defers to the agent's shared detector _is_personal_recall.
  • Refusals, errors, and low-confidence answers. Answers that start with "error"/"unavailable", contain "trouble connecting"/"system issue", carry grounding disclaimers, or are refusal phrases like "I do not have"/"I cannot access" are not stored. Caching a refusal would keep serving it even after the bot is fixed.
Note: In local development there is usually no Redis configured, so REDIS_URL is blank, the L2 connection "fails" gracefully, and only L1 (and Semantic, if the embedding client is available) is used. Setting REDIS_URL turns L2 back on. The raw cache key is built from the normalized question plus a small model-name hash, which is exactly why personal-recall answers are excluded from caching entirely rather than relying on the key.

6.5 Authentication & roles (knowing who you are, and what you may do)

Authentication is proving who you are; authorization is what you're allowed to do once known. ORA Navigator handles both with email-restricted signup, email verification, hashed passwords, and JWT tokens.

Signup is restricted to @morgan.edu, with a "verify first" flow

The auth endpoints live in backend/routers/auth.py. Signup (POST /api/register) does these checks: the email format must be valid; at most 3 signup attempts per email per hour (returns HTTP 429 if exceeded); only addresses ending in morgan.edu are allowed (a test-only @test.com exception is gated behind ALLOW_TEST_EMAILS, which is off in production); and the password must be at least 8 characters.

Crucially, /api/register does not create a user row. Instead it builds a signed JWT that contains the email, the already-hashed password, and the name, and emails that token as a verification link. The user row is only created when the person clicks the link and GET /api/verify-email decodes the token and inserts the user with email_verified=True. So if someone never clicks, no account ever exists.

Login, and why admins skip verification

POST /api/login looks up the user by email, checks the password, and — if the password is right — issues a JWT access token. There's a guard that rejects unverified accounts, but it explicitly exempts admins: if not email_verified and role != "admin". The admins are milam5@morgan.edu (user id 6) and a backup admin@local.dev; the admin account is created/upgraded on startup using the ADMIN_EMAIL and ADMIN_PASSWORD environment values.

Password hashing (passwords are never stored in plain text)

Passwords are run through bcrypt, a deliberately slow one-way hashing algorithm, in backend/security.py. hash_password truncates the password to bcrypt's 72-byte limit, generates a random "salt" with 12 rounds of work, and stores only the resulting hash. verify_password re-hashes the entered password and checks it against the stored hash. Because hashing is one-way, even someone with the database cannot read the real passwords.

JWT tokens (the "I'm logged in" badge)

A JWT (JSON Web Token) is a signed string the browser keeps after login and sends on each request to prove identity. create_access_token packs the user's id, role, and email into the token, adds an expiry (default 240 minutes), and signs it with the secret JWT_SECRET using the HS256 algorithm. On every protected request, get_current_user in backend/deps.py decodes the token with that same secret, loads the matching user, and returns a small dict of user_id, email, role, name, and the profile fields. If the token is invalid or the user is missing, it raises HTTP 403. A sibling helper get_optional_user returns None instead of erroring, for endpoints that also serve guests.

Where the secrets come from

JWT_SECRET is read from the environment, and the code refuses to start if it's missing. In production these values come from Google Secret Manager — for example ora-jwt-secret (the JWT signing key), plus ora-database-url, ora-admin-email, ora-admin-password, and ora-redis-url. Storing secrets in Secret Manager (instead of in the code) means they can be rotated without changing code, and they never appear in the repository.

Note (the ProfileUpdateRequest lesson): FastAPI uses small classes called request models to describe and validate the JSON a client sends. backend/deps.py is the single home for these. There must be exactly one definition of each request model. Earlier, a duplicate local copy of ProfileUpdateRequest in main.py silently "shadowed" the real one, so newly added profile fields were quietly dropped. Keeping these models defined only in deps.py ensures every endpoint sees the same, complete set of fields.
