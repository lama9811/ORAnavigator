7. Cron Jobs & All Scheduled Work

This section explains every piece of work in ORA Navigator that runs by itself on a timer, plus the small background jobs that fire automatically right after every chat message. Nobody clicks a button to start these; the computer runs them on its own.

7.1 Two quick definitions

  • Cron job — a task that runs automatically on a fixed schedule (for example, "every day at 3 in the morning"). The name comes from an old Unix program called cron. You set the schedule once and it keeps firing forever.
  • Cloud Scheduler — Google Cloud's service that acts as the alarm clock. At the scheduled time it sends an HTTP request (it "pings" a web address) to wake up the job. ORA Navigator's scheduled jobs are Cloud Scheduler jobs that POST to a special "internal" web address on the backend.

So the pattern is always the same: Cloud Scheduler (the alarm) → calls an internal endpoint on the backend (the door) → which runs a Python function (the actual work) → which writes results into a database table.

7.2 How the alarm proves it is allowed in (the auth model)

The internal endpoints are not for normal users. They are protected by a shared secret — basically a long password. Every internal endpoint reads an HTTP header called X-Research-Secret from the incoming request and compares it to an environment variable called RESEARCH_SECRET on the server. If they do not match exactly, the request is rejected with HTTP 403 ("forbidden").

The code is the same few lines for every internal endpoint: read the header, read the env var, and if the env var is blank or the two do not match, raise a 403.

Note: Both Cloud Scheduler jobs (the 7am deadline job and the 3am memory job) use the same shared secret, sent in the same X-Research-Secret header. This secret was rotated (changed to a fresh value) on 2026-06-02 after it was briefly exposed; the value stored in Cloud Run and the value baked into both scheduler job headers were re-synced and verified to match.
Note (local-development gotcha): When testing this stack on a laptop, the local ADK agent must be started with python -m google.adk.cli api_server . --port 8081. The friendlier adk web developer-UI mode crashes here on a uvicorn config.load() error; api_server is the plain, headless mode that the backend actually talks to.

7.4 The two real Cloud Scheduler cron jobs, in detail

(1) ora-deadline-watcher — daily 7am Eastern

This is the proposal-deadline reminder system. The chain is: Cloud Scheduler → POST /api/internal/deadlines/check → send_due_reminders().

  • It defines five "reminder buckets" — THRESHOLD_DAYS = (14, 7, 3, 1, 0). A faculty member gets pinged when their proposal deadline is 14, 7, 3, 1, and 0 days away.
  • find_due_reminders() scans active proposals and keeps only the ones whose deadline lands exactly on one of those buckets today.
  • For each one, it first tries to write a friendly, personalized email body with Gemini. If the AI is slow or fails, it falls back hard to a fixed template — so a reminder always sends.
  • It emails the proposal's owner.

What it writes + where: only after a successful send, it inserts one DeadlineReminderLog row into the deadline_reminder_log table, recording the submission, the threshold bucket, and who it was sent to. This row is the "I already sent this one" memory. Because of it, if the job is run twice, the same person is never double-emailed for the same bucket — this is called idempotency. If the send fails, no log row is written, so the next run retries.

(2) ora-memory-consolidate — daily 3am Eastern

Chain: Cloud Scheduler → POST /api/internal/memory/consolidate → consolidate_user_memories(24).

  • It looks at the last 24 hours of chat and finds every user who has messages in that window.
  • For each such user it grabs up to 50 recent messages and builds a transcript, but only if they have at least 3 messages — very quiet users are skipped.
  • It asks Gemini to extract durable one-line facts about the user's research-admin context — things like their role, department, or active grant. The prompt deliberately excludes private/financial details.
  • It merges those facts in with _merge_memories().

What it writes + where: fact rows in the user_memories table. The merge logic is careful: a new fact is skipped if an existing fact already contains it, or vice-versa (dedup); and if a category already has 5 facts, it overwrites the oldest one of that type instead of growing without limit (cap of 5 per type).

Note (very important): This 3am job only reads chat_history — it never deletes or edits chat transcripts. The word "consolidate" means it distills and caps the fact list in the separate user_memories table, not "clean up the chat." Your raw conversation history is left completely untouched.

7.5 The idle-sweep cron — every 5 minutes

Chain: Cloud Scheduler (every 5 min) → POST /api/internal/memory/idle-sweep → consolidate_idle_users(5, 10).

The 3am job catches everyone overnight, but that means a fact you mention at 10am could sit uncaptured for many hours. The idle-sweep closes that gap. Every 5 minutes it queries the users table for anyone whose last_chat_at is between 5 and 10 minutes ago — meaning they just stopped chatting. For each of those users it runs the same realtime extraction (over the last 2 hours) so their mid-session facts get saved before they ever reach the 3am cron. Re-running is harmless — re-extraction just rewrites the same facts and the dedup logic absorbs the repeats.

7.6 The research batch — turning failed questions into draft answers

Chain: POST /api/internal/research/run (and the admin dashboard's "Run research now" button) → run_research_batch().

This is the self-improvement loop for the knowledge base. In plain words:

  • Cluster: cluster_failed_queries() takes questions the bot answered poorly, turns each into an embedding, and greedily groups similar ones together by cosine similarity so "How do I add a co-PI?" and "adding a co-investigator?" land in one cluster.
  • Research: for the biggest clusters it picks a representative question and calls research_topic() to draft a researched answer with sources and a confidence level.
  • Suggest: it creates a KBSuggestion row with the topic, the drafted answer, the sources, and a status of "pending", then marks the cluster's queries as "researched".

These suggestions are proposals for a human admin to approve, reject, or push into the real knowledge base — the bot never silently adds them itself. (See Section 4 for the full five-stage lifecycle.)

7.7 Per-turn background work (not cron, but automatic)

Cron jobs run on a clock. This last group runs on an event instead: it fires automatically right after every chat message is saved. All of it is launched from one helper, _schedule_post_commit_memory_tasks(), called from each chat endpoint after the answer is already committed. The key idea: this work happens after the response has already been sent to the user, so it adds zero waiting time to the chat. Each sub-task is independently switchable with a feature flag and self-checks whether it should actually do anything.

The five sub-tasks it fires:

  • Session summary: summarizes the earlier turns of a long chat (self-gates to ~8+ turns). Writes into chat_history.session_summary.
  • Touch last-chat time: stamps users.last_chat_at = now(). This is the breadcrumb the 5-minute idle-sweep cron reads to know who just went quiet.
  • Embed turn: converts the question + answer into a numeric embedding for future cross-session semantic recall. Writes the embedding onto the chat_history turn.
  • Realtime memory extraction: counts the session's turns; only every 6th turn does it actually run. When it fires, it pulls durable facts from the last ~2 hours and merges them. A per-user lock stops two of these from running at once for the same person. Writes fact rows into user_memories.
  • Suggestion regeneration: refreshes the personalized home-screen question suggestions, throttled internally to a 10-minute window. Writes rows into user_suggested_questions.
Note: None of these per-turn tasks use the X-Research-Secret header — they are not web endpoints. They run as fire-and-forget background tasks inside the backend process, so there is no external request to authenticate. The secret-protected endpoints are only the cron-triggered ones in Section 7.3.
