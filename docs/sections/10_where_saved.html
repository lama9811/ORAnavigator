10. "Where Is It Saved?" — Master Reference Table

This is a quick-lookup table for one of the most common questions: when something happens in ORA Navigator, where does that piece of data physically end up? Each row names one artifact (a thing the system produces or stores), exactly where it lives, and how long it stays there. "Cloud SQL" is the MySQL database; "Vertex AI Search datastore" is the searchable knowledge-base index; "in memory only" means it exists just for the few seconds of one web request and is then gone.

Artifact (the thing)Where it is savedLifetime / notes
A user accountCloud SQL → users table (one row)Permanent until deleted. Created only after email verification.
A chat question + answer (one turn)Cloud SQL → chat_history table (one row)Permanent. Raw transcript is never auto-deleted (the 3am job only reads it).
The "Sources" links under an answerCloud SQL → chat_history.citations column (JSON), and a parallel cache entry under a cit: keyPermanent in the DB; the cache copy expires with the cached answer. This is why Sources survive refresh/cache/history.
A cached answer (to skip re-asking the AI)L1 in-memory (24h) → L2 Redis (7 days) → Semantic cacheTemporary. Refusals and personal-recall answers are never cached.
A durable fact about a user (role, dept, grant…)Cloud SQL → user_memories table (one row per fact)Permanent, but capped at 5 per type — a 6th overwrites the oldest. User can pause or delete.
A rolling session summaryCloud SQL → chat_history.session_summary column on the newest turnPermanent; rebuilt and extended as the conversation grows.
A turn's meaning-vector (for cross-session recall)Cloud SQL → chat_history.embedding columnPermanent; computed in the background after the turn is saved.
Your personalized "suggested questions"Cloud SQL → user_suggested_questions table (one row per user)Permanent; regenerated in the background after chats (throttled to ~10 min).
A question the bot failed to answerCloud SQL → failed_queries table (one row)Permanent; moves through statuses new → clustered → researched.
A drafted KB fix (from the research agent)Cloud SQL → kb_suggestions table (one row)Permanent; pending → approved/rejected → pushed.
A published KB answer (after admin "Push")Vertex AI Search datastore oranavigator-kb-v8 (appended to or created as a document)Permanent and live — the chatbot can now find it. Caches are cleared on push.
The 382 source KB documentsOn disk → backend/kb_structured/ (one .json file each)Permanent; the source of truth that is indexed into the datastore.
An uploaded solicitation PDFIn memory only — read, parsed, then discardedNever saved to disk or any table. Only the confirmed extracted facts persist.
The extracted solicitation fields (before you confirm)Nowhere — returned in the HTTP response, held in the browserGone unless you click Confirm.
A confirmed proposalCloud SQL → submissions table (one row); solicitation rules live in submissions.notesPermanent until the user deletes it.
A proposal checklist itemCloud SQL → submission_tasks table (one row)Permanent; cascade-deletes with its proposal.
Saved Budget Helper inputs / Compliance answers / Coach draftsCloud SQL → submissions.budget_json / compliance_json / sections_json (JSON text)Permanent with the proposal; these power the has_budget / has_compliance / has_sections pathway badges.
A computed budget / justification / opportunity searchNowhere by itself — computed on request and returned to the browserThe Budget Helper math, the AI justification, and the Opportunity Finder's Grants.gov results are all live; only the inputs you save (above) persist.
An uploaded draft proposal PDF (for critique), or a fetched solicitation URL's textIn memory only — checked, then discardedNever stored. Only the critique result / extracted facts are returned to the browser.
The authored sample-proposal PDFsOn disk → backend/sample_proposals/ (committed to the repo)Permanent; generated by a dev script and shipped inside the Docker image so the public download works.
A sent deadline reminderCloud SQL → deadline_reminder_log table (one row per email)Permanent; this is the record that prevents double-sending.
A support ticket (+ its optional attachment)Cloud SQL → support_tickets table (attachment as base64 in the row)Permanent until closed/deleted.
Thumbs-up/down/report feedbackCloud SQL → feedback table (one row); a "report" also writes a failed_queries rowPermanent.
A user's profile (Department/Title/Role/Interests)Cloud SQL → users columns, and mirrored into user_memoriesPermanent; profile values overwrite auto-learned facts.
Secrets (DB password, JWT key, admin login, Redis URL, Firecrawl key)Google Secret Manager (read into env vars at startup)Permanent; never written in code or the repo. Rotatable. The optional Firecrawl key self-heals to a placeholder if missing.
The built application (the running programs)Artifact Registry (container images) → run on Cloud RunEach deploy pushes a new image.
The one thing to remember: almost everything durable lives in Cloud SQL (the MySQL database). The two big exceptions are the knowledge base (source files on disk + the searchable Vertex datastore) and uploaded PDFs (never stored at all — read in memory, used, discarded). Secrets live in Secret Manager; temporary speed-ups live in the cache.
